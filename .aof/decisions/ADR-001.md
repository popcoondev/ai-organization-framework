# ADR-001: Adopt Organization Operating System Direction For The Next Major Version

Status: accepted

Date: 2026-06-12

## Context

AOF v2.0 established the installer, bootstrap, and upgrade path. AOF v2.1 established `.aof/organization.json`, Role-Agent separation, and the baseline organization schema.

The next major version needs to prove that AOF can operate as an organization planning system, not only as documentation or agent execution guidance.

## Decision

Use AOF 2.1 itself to plan the next major version through:

- Requirements analysis
- Organization formation
- Team charters
- Contract definition
- ADR creation
- Council approval
- Roadmap creation
- Release plan creation

This benchmark intentionally stays inside the current AOF 2.1 artifact model. Runtime enforcement, backend-specific orchestration, and autonomous agent execution remain later implementation scope unless explicitly approved.

## Council Approval

Approval policy: `2_of_3`

Outcome: approved

| Council | Vote | Rationale |
| --- | --- | --- |
| Product Council | approve | The direction ties directly to user value by proving that AOF can operate as an organization system. |
| Architecture Council | approve | The plan respects the v2.1 model, preserves Role-Agent separation, and does not assume runtime features that are not implemented. |
| Operations Council | approve | The phased roadmap contains verification gates and defers autonomous runtime claims until implementation evidence exists. |

Guardian veto: none

## Consequences

- The next major version is planned around organization operation, not agent execution alone.
- Implementation must preserve Role-Agent separation.
- Council approval is represented as a decision artifact, not as a team dependency.
- Runtime enforcement, metrics automation, and backend-specific orchestration remain later scoped phases.
