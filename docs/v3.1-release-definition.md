# AOF v3.1 Release Definition

`v3.1.0` の主題は、`v3.0.x` で成立した self-hosting runtime loop を、
release-grade benchmark coverage と multi-family runtime evidence に拡張することである。

## Position

`v3.0.0` は、AOF が backend-neutral organization runtime として
1 本の auditable proof loop を self-hosting repo 上で成立させられることを示した。  
`v3.1.0` では、その loop が

- weak artifact を見逃さない review runtime
- runtime bypass を許さない orchestration discipline
- 追加 backend family でも崩れない proof shape

を持つことまでを release gate に含める。

ここでやるのは autonomous workforce management claim ではない。  
やるのは、`v3.0.x` の proof loop を
**benchmarkable, repeatable, broader-than-one-family**
にすることである。

## v3.1 Goals

1. review-validity benchmark を runnable artifact set と recorded verdict 群まで operationalize する
2. orchestrator runtime-discipline benchmark を reusable negative / positive path まで operationalize する
3. current runtime proof を追加 backend family に拡張し、single-family dependence を release blocker として扱わない
4. runtime correctness matrix と benchmark execution status を、`v3.1.0` gate judgment に使える状態にする
5. `v3.0.x` の baseline green を壊さずに、coverage expansion を first-class release scope にする

## Out Of Scope

- always-on autonomous scheduler claims
- adaptive organization redesign without explicit human approval
- provider-native child launcher automation
- generalized workforce analytics beyond the current benchmark artifacts
- productized multi-provider parity claims beyond the recorded proof families

## Release Gates

### Gate 1: Review Validity Benchmark Is Operationalized

- review-validity case set exists as committed runtime artifact
- weak-artifact rejection is recorded as an auditable verdict family
- human-low / ai-high conflict is recorded as an auditable verdict family
- runtime-generated reopen or reevaluation traces exist for the benchmark path
- review packet structure is sufficient for human-facing quality rejection, not only internal AI approval

### Gate 2: Runtime Discipline Benchmark Is Operationalized

- prose-only orchestration is rejected by committed benchmark evidence
- partial-runtime-use chains are rejected by committed benchmark evidence
- latest positive runtime path can be replayed through one command
- positive runtime audit path emits machine-readable audit note, audit packet, and reconstruction map
- runtime bypass or missing chain evidence is treated as a blocking benchmark failure

### Gate 3: Runtime Coverage Expands Beyond One Proof Family

- current runtime proof remains green on the self-hosting repo
- at least one additional backend family or proof family is recorded as follow-through evidence for the same auditable loop shape
- runtime-coverage matrix no longer depends on a single-family execution claim for release honesty
- `TASK-034` is resolved or replaced by stronger equivalent recorded evidence

### Gate 4: Baseline Runtime Safety Remains Green

- `organization-audit --project .` remains green
- `organization-verify --project .` remains green
- `runtime-loop-proof --project . --provider mock --source-task-id TASK-011` remains green
- `runtime-discipline-benchmark --project . --source-task-id TASK-011` remains green
- `npm test -- --runInBand` remains green
- `npm run smoke` remains green

### Gate 5: Boundary Remains Honest

- `v3.1.0` claims stronger runtime correctness and benchmark coverage, not solved autonomy
- `v3.1.0` does not claim human review is unnecessary
- `v3.1.0` does not claim all runtime stages are fully benchmark-complete if the coverage matrix still says partial or weak
- `v3.1.0` does not claim broad production provider parity beyond the recorded evidence set

## Current Blocking Set

2026-06-16 時点では、少なくとも次が `v3.1.0` の release blocker である。

1. `TASK-035`: review-validity benchmark の broader family coverage
2. `TASK-036`: runtime-discipline benchmark の broader audit automation と bypass detection coverage
3. `TASK-034`: additional backend family への runtime proof 拡張
4. `runtime-coverage` matrix 上の `framing`, `allocation`, `runtime-stage-to-benchmark mapping` の partial state

## Intended Outcome

After `v3.1.0`, AOF operator は self-hosting repo 上で次の判断を
release-grade evidence として扱える。

```text
one proof loop exists
  -> benchmark families can replay failure and pass conditions
  -> human-low / ai-high conflict is recorded and reopenable
  -> runtime bypass is structurally rejectable
  -> positive audit path is reconstructable from machine-readable artifacts
  -> the same loop shape is evidenced beyond one proof family
```

この版の価値は、新しい autonomy claim ではなく、  
`v3.0.x` の loop proof を
**minor release として信頼してよい benchmark surface**
まで引き上げる点にある。
