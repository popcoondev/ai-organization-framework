# AOF v3.9 Release Definition

`v3.9.0` の主題は、
**operator brief の下に progress / tree position / evidence drill-down を置き、viewer までを 1 本の runtime path にすること**
である。

## Position

`v3.8.0` で runtime は one-packet operator brief を返せるようになった。  
しかし実 runtime で viewer を見ると、まだ次の gap が残っていた。

- current answer は読める
- ただし `何が変わったか` が弱い
- `どの trunk / branch にいるか` が弱い
- `なぜそれが正しいか` の proof path が surface として浅い
- browser / viewer がまだ manual ritual で、runtime path が途中で切れる

したがって `v3.9.0` でやるべきことは decorative dashboard ではない。  
やるべきことは **brief の下に bounded operator progress layer を作り、viewer session まで runtime に含めること** である。

## v3.9 Goals

1. a first-class `operator-progress` command exists
2. a first-class `tree-position` command exists
3. a first-class `evidence-drill-down` command exists
4. `visibility-export` emits those packets from the same canonical runtime truth
5. a first-class `visibility-session` command exists
6. the viewer can show `Now`, `What Changed`, `Where In The Tree`, and `Evidence Drill-Down` from one exported packet

## Release Gates

### Gate 1: Progress Surface Exists

- a dedicated `operator-progress` command exists
- the progress packet explains what changed since the last meaningful checkpoint
- the packet stays tied to the current frontier task

### Gate 2: Tree Position Exists

- a dedicated `tree-position` command exists
- the tree packet states the current shipped trunk and the live frontier branch
- the tree packet stays aligned with the release and roadmap truth

### Gate 3: Proof Surface Exists

- a dedicated `evidence-drill-down` command exists
- the packet explains why the current headline, blockers, and next action are true
- the packet stays bounded to canonical runtime refs instead of becoming a second narrative

### Gate 4: Viewer Session Becomes One Runtime Path

- `visibility-session` exports the packet and starts the viewer session
- browser opening can be triggered from the same runtime command
- operators no longer need to manually stitch export + serve + browser open just to see the upgraded surface

### Gate 5: Viewer Shows Real Progress

- the viewer can show `What Changed`
- the viewer can show `Where In The Tree`
- the viewer can show `Evidence Drill-Down`
- the release creates a visible upgrade over the mostly static v3.8 surface

### Gate 6: Boundary Remains Honest

- progress / tree / proof packets remain derived from canonical runtime artifacts
- the viewer does not become the primary truth source
- Need Validation remains approval authority
- Discovery remains evidence

## Intended Outcome

After `v3.9.0`, the runtime should be able to give the operator one coherent path:

```text
this is what is happening now
this is what changed since the last checkpoint
this is where this work sits in the larger release tree
this is the bounded proof path for the current answer
and this whole surface is reachable from one runtime command
```
