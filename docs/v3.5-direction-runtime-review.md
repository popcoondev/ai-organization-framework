## Question

What should AOF do next after `v3.4.0` if the goal is to improve real operator performance without breaking current governance boundaries?

## Proposal Under Review

The reviewed proposal argues that the next problem is not command insufficiency first, but **CLI context efficiency**:

- commands are numerous enough that full-reference rereads are expensive
- command purpose is partly inferable, but not formally classified
- AI and humans both pay routing cost before they can use the runtime correctly

## Validated Need

AOF needs a canonical command-routing layer that classifies the CLI surface and exposes minimal registry metadata through the AI recognition packet.

This is narrower and better than the raw request "improve CLI context efficiency" because it identifies:

- the actual bottleneck: routing and discoverability
- the likely product surface: command taxonomy + registry + packet integration
- the non-goals: command-semantic rewrite and authority expansion

## Why This Is A Good Next Goal

- It builds on `v3.4.0` rather than competing with it.
- It improves a real post-release operator pain point.
- It can be implemented as additive metadata and packet design rather than disruptive runtime redesign.
- It preserves Discovery / Need Validation / backend-neutral boundaries.

## Risks

- A non-canonical registry could become duplicate documentation.
- Taxonomy categories may be too vague or too granular.
- The routing layer must prove actual selection value, not just add another file.

## Recommendation

Proceed with:

- `v3.5.0 = CLI Context Efficiency Layer`

Core release claims should be limited to:

- command taxonomy
- command registry artifact
- AI recognition-packet routing integration
- verification that major operator paths can be discovered from the new layer
