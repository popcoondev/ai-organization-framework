# AOF v1.4 Human Visibility Validation

`v1.4` の Human Visibility Layer が、viewer 実装なしでも current state explanation と runtime distinction を支えられるかを確認するための concise validation artifact。

## Validation Target

この validation は、`candidate lifecycle` を使った partial runtime を対象にする。  
プロダクト例としては `日替わり観測候補` の生成 / 選定 / 配信を想定する。

## What Must Be Proven

`v1.4` では次の 3 つが読めることを確認する。

1. 人間が **今どこにいるか** を `status_card` で把握できる
2. 人間が **誰が何を決めてこうなったか** を `timeline_feed` で追える
3. 人間が **現在の phase と次の phase** を `flow_snapshot` で理解できる

加えて、`runtime-used AOF` と `static AOF` の区別が人間から見えることを確認する。

## Event Basis

この validation で使う event basis は次の 3 つである。

- `candidate_generated`
- `candidate_selected`
- `candidate_published`

各 event の field contract は [docs/candidate-lifecycle-event-contract.md](./candidate-lifecycle-event-contract.md) を正本とする。

## Validation Scenario

scenario:

- `candidate_generated`
  - 候補群が作られた
- `candidate_selected`
  - 今日の候補が 1 つ選ばれた
- `candidate_published`
  - 外部に出す候補が更新された

人間が知りたいのは、

- いま publish 前か後か
- 何を根拠に今日の候補が選ばれたか
- 次に何を確認するのか

である。

## Output Checks

### 1. Status Check

`status_card` が次を示せること。

- current phase
- current operating goal
- owner
- open signals
- next checkpoint
- runtime evidence state

判定:

- `candidate_selected` の時点なら、
  - `current_phase = candidate_selected`
  - `runtime_evidence_state = present`
  - `next_checkpoint = publish verification`
  が読める

### 2. Timeline Check

`timeline_feed` が次を示せること。

- when
- who
- what changed
- why it changed
- what became next

判定:

- `candidate_selected` entry で
  - actor
  - rationale
  - next
  が読める

### 3. Flow Check

`flow_snapshot` が次を示せること。

- `candidate_generated -> candidate_selected -> candidate_published`
- current node
- pending next step

判定:

- `candidate_selected` が current
- `candidate_published` が pending
  として読める

## Static vs Runtime-Used Check

次の差が人間に見えることを確認する。

### Static AOF

- decision records はある
- しかし event basis が無い
- `runtime_evidence_state = none`

### Runtime-Used AOF

- event basis がある
- rationale と next checkpoint がある
- `runtime_evidence_state = present`

この差が `status_card` 上で一目で分かるなら、visibility layer は最低限成立している。

## Result

現時点の docs だけでも、`candidate lifecycle` を基礎に

- status
- timeline
- flow
- static vs runtime-used distinction

を一貫した shape で表現できる。  
したがって `v1.4` は、viewer 実装前でも **Human Visibility Layer の contract level validation** を主張できる。
