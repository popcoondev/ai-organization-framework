**release-state freshness and drift detection for the self-hosting runtime**

## What The Runtime Exposed

### 1. Active operator surfaces disagreed on the current release baseline

Observed through runtime commands:

- `organization-status --project .`
- `contract-register --project .`
- `roadmap-status --project .`

Reading:

- `organization-status` reported `v3.3`-aligned operating goals.
- `contract-register` still pointed the governance-to-release contract at `docs/v3.0-release-definition.md`.
- `roadmap-status` still reported `docs/v3.0-release-definition.md` as the current release definition.

This is not a cosmetic mismatch.  
It means operator-facing runtime inspection cannot be trusted to represent the current governed baseline after a release.

### 2. Bootstrap and organization metadata were still historical

Observed in:

- `.aof/project-bootstrap.json`
- `.aof/organization.json`

Reading:

- `.aof/project-bootstrap.json` still reported `aof_version: 2.2.0`
- `.aof/organization.json` still framed the mission around packaging AOF honestly as `v3.0`

This means the self-hosting runtime has no canonical mechanism for rolling active release-state forward after a release.

### 3. Manual docs and release tagging were not enough

`v3.3.0` was actually released and tagged, but active runtime artifacts still drifted.

Therefore the next problem is not:

- more autonomy
- more discovery breadth
- more backend integration

The next problem is:

- making the runtime state truthful after release

## Direction

`v3.4.x` should focus on:

1. one canonical active release manifest
2. release-state drift detection across operator-facing runtime surfaces
3. a runtime-native refresh/update path for active release refs
4. explicit separation between active refs and intentionally historical refs

## Proposed Scope

Candidate `v3.4` outcomes:

- `active-release-manifest` artifact family
- release-state audit command or benchmark
- release-state refresh command
- verification that `organization-status`, `contract-register`, `roadmap-status`, and bootstrap metadata all resolve through one active release baseline

## Non-Goals

`v3.4` should not:

- weaken Need Validation
- broaden backend-specific autonomy claims
- rely on manual JSON editing as the primary operating method
- collapse historical release references into active release pointers without policy

## Why This Is The Highest-Leverage Next Step

After `v3.3.0`, AOF already has:

- discovery evidence contract
- discovery judgment contract
- discovery handoff into Need Validation

What it does not yet have is a trustworthy active release-state layer for the self-hosting runtime itself.

If this is not fixed next, later roadmap work will continue to inherit stale operator surfaces and weak runtime truthfulness.

## Recommendation

Adopt this as the next evolution point:

**AOF v3.4 = Release-State Freshness And Drift Detection**
