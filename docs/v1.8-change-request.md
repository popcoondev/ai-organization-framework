# AOF v1.8 Change Request

- 作成日: `2026-06-02`
- 作成者: レビュー分析（claude/framework-review-HCxns）
- 対象バージョン: `v1.8`
- 前提: `v1.7.0` リリース済み

## Position

`v1.7` が「AI Orchestrator を第一級アクターとして定義し、並列スレッド統治モデルを formalize する版」であるなら、  
`v1.8` は **「AOF がプロジェクト全体の記憶を自己保有し、タスクとライフサイクルを AOF ネイティブに管理する版」** である。

### 核心的再認識

`v1.0` 〜 `v1.7` を通じて、AOF は以下のギャップを暗黙的に回避していた。

```
[発見されたタスク]
  └── Discovery が発見 → GitHub Issues（外部）に起票
  └── AOF が内部で保持する仕組みがない
  └── 古いタスクの棚卸しルールがない
  └── セッション完了後の圧縮・アーカイブルールがない

[Orchestrator のプロンプト・中間ファイル]
  └── セッション開始のたびに再構成（定義なし）
  └── スレッド間の中間出力の置き場がない
  └── .aof/ に収まっていない部分が多い
```

`AOF` が「組織の記憶を再現性をもって保持する」ことを謳うなら、  
**タスク・ライフサイクル・Orchestrator プロンプトを .aof/ に一元化する必要がある。**

また、このギャップが `v1.7` まで発見されなかった根本原因は、  
**AOF がフレームワーク自身に Discovery プロセスを適用していなかったこと** にある。  
`v1.8` は AOF の自己監査機構（Framework Self-Audit）を導入し、この再発を防ぐ。

## 変更 1: AOF ネイティブタスク管理（新規）

### 1-1. `.aof/tasks/` ディレクトリ（新規）

`.aof/tasks/` を AOF の唯一のタスク台帳とする。

```
.aof/
  tasks/
    open/          # 未着手・着手待ちタスク
    assigned/      # 進行中（担当セッション ID 付き）
    done/          # 完了済み（アーカイブ前）
    archived/      # 棚卸し済み（過去ログ）
    retired/       # 破棄確定（実施しないと決定）
```

各タスクは JSON ファイル 1 件とする（`TASK-{id}.json`）。  
スキーマは「変更 4: スキーマ追加」で定義する。

#### タスクのライフサイクル

```
Discovery フェーズ
  → open/       （Discovery Scout が起票）
       ↓
  Triage（Alignment Pulse 契機）
       ↓              ↓
  assigned/       retired/
  （着手決定）    （棚卸し除外）
       ↓
  done/
       ↓
  Alignment Pulse 契機または Archivist
  → archived/
```

#### タスクの起票者

- Discovery Scout（フェーズ実行中）
- Experience Steward（不整合検出時）
- Guardian（リスク検出時）
- Orchestrator（Human 指示受領時）
- Human（手動追加）

#### 外部連携

- GitHub Issues との連携は `v1.8` の scope 外とする
- GitHub Issues への export は v1.9 以降のオプションとする
- `.aof/tasks/` が source of truth であり、外部ツールは mirror とする

---

### 1-2. `docs/task-management-model.md`（新規ドキュメント）

以下を定義する。

- `.aof/tasks/` のディレクトリ構造と命名規則
- タスクライフサイクル（Discovery → Open → Triage → Assigned → Done → Archive/Retired）
- 各ライフサイクル遷移の trigger 条件と権限
- タスク起票ルール（誰が、いつ、何を起票するか）
- タスクと Session の紐付け（assigned セッション ID）
- タスクと Decision Record の紐付け（完了した Task に関連する Decision Record refs）

## 変更 2: Alignment Pulse の拡張

### 2-1. `docs/value-alignment-loop.md`（更新）

Alignment Pulse の定期確認項目に **タスク棚卸し（Task Triage）** を追加する。

現行（v1.7 で定義予定の 3 トリガー後）に 4 番目の確認項目として追加する。

| Alignment Pulse 確認項目 | 内容 |
|---|---|
| 1. Goal Alignment | North Star Goal / Operating Goal の整合確認 |
| 2. Open Mismatch | Experience Steward 検出の未解決不整合確認 |
| 3. Guardian Veto Pending | Guardian の未解決リスク確認 |
| 4. Task Triage（v1.8 新規） | `.aof/tasks/open/` の棚卸し：Operating Goal との整合・優先順位・retire 判定 |

**Task Triage の具体的な問い（Orchestrator → Human）**

- 「現在 N 件の未着手タスクがあります。Current Operating Goal との整合で優先順位を確認してください」
- 「X 日以上更新されていないタスクがあります。retire しますか？」
- 「新しい Value Slice の着手前に、関連する open タスクを確認してください」

## 変更 3: Archivist ロールの拡張

### 3-1. `docs/context-lifecycle-model.md`（更新）

Archivist ロール（既存）の責務を拡張し、タスク管理を明示する。

#### 現行 Archivist の責務

- セッションコンテキストの圧縮（Active → Summary）
- アーカイブ判定と実施（Summary → Archived）
- コンテキストウィンドウの適切な維持

#### v1.8 追加責務

- `.aof/tasks/done/` → `.aof/tasks/archived/` への定期移動
- 古いタスク（retire 候補）を Alignment Pulse に持ち込むための準備
- セッション完了後のスレッドローカル評価の圧縮・アーカイブ
- `.aof/context/threads/` 内の完了スレッド出力の整理

#### Archivist のトリガー条件

- Alignment Pulse 契機（定期実行）
- Orchestrator の明示呼び出し（大型スライス完了後など）
- Human の手動指示

## 変更 4: `.aof/` ディレクトリ構造の整備

### 4-1. `.aof/` 完全構造定義（新規ドキュメント）

`docs/aof-directory-structure.md` を新規作成する。

定義する `.aof/` 構造：

```
.aof/
  sessions/              # セッション状態ファイル（既存）
  decision-records/      # Decision Record（既存）
  context/
    active/              # アクティブなコンテキスト要約（既存相当）
    threads/             # 並列スレッドの中間出力置き場（v1.8 新規）
    archived/            # 圧縮済みコンテキスト（既存相当）
  tasks/                 # AOF ネイティブタスク台帳（v1.8 新規）
    open/
    assigned/
    done/
    archived/
    retired/
  prompts/               # Orchestrator・スレッド用プロンプトテンプレート（v1.8 新規）
    orchestrator/
    council/
    discovery/
    steward/
  goals/                 # Goal 状態ファイル（v1.8 新規）
    north-star.json
    operating-goal.json
    next-value-slice.json
```

#### `.aof/context/threads/` の用途

並列スレッドが実行中に生成する中間ファイルを置く。

- Council スレッドの評価草稿（Orchestrator 集約前）
- Discovery Scout の探索ログ
- Experience Steward の評価メモ
- 各スレッド完了後は Archivist が `.aof/context/archived/` へ移動

#### `.aof/prompts/` の用途

Orchestrator がスレッドを起動するときに使うプロンプトテンプレートを置く。

- セッション開始のたびに再構成するのではなく、バージョン管理された定型文として保持する
- テンプレートには `{session_id}`, `{goal}`, `{context_summary}` 等のプレースホルダーを使う
- 更新は Orchestrator または Human が行う

#### `.aof/goals/` の用途

現在の 3 Goal Layer の状態を JSON ファイルとして保持する。

- `north-star.json`: North Star Goal（Orchestrator が更新、Human 合意必須）
- `operating-goal.json`: Current Operating Goal（Orchestrator が更新）
- `next-value-slice.json`: Next Value Slice（Builder 完了後に Orchestrator が更新）

v1.7 の State Ownership Model と整合する形で実装する。

## 変更 5: フレームワーク自己監査機構

### 5-1. `docs/framework-self-audit.md`（新規ドキュメント）

AOF 自体の Discovery を定期的に実施するための仕組みを定義する。

#### 背景

`v1.7` まで、タスク管理ギャップが発見されなかった根本原因は以下である。

1. レビューが各バージョンのゲート確認に閉じており、「AOF 全体として何が欠けているか」の問いを立てなかった
2. GitHub Issues への依存が暗黙の前提になっており、外部ツールへの依存自体を問題として認識しなかった
3. AOF 自体に Framework Discovery プロセスを適用していなかった

#### 自己監査の定義

AOF の主要リリースごと（または半年ごと）に、AOF 自身を対象とした Discovery フェーズを実施する。

- **問い**: 「AOF はプロジェクトのどの側面を管理できていないか？」
- **担当**: Orchestrator（Human 要求なしに定期実行）
- **出力**: `.aof/tasks/open/TASK-{id}.json`（発見された欠如を起票）
- **Human へのエスカレーション**: 設計変更を伴う発見のみ

#### 自己監査のスコープ

| 領域 | 確認する問い |
|---|---|
| ライフサイクル | `.aof/` 内の各アーティファクトに保持・廃棄ルールがあるか |
| 外部依存 | AOF が外部ツールに暗黙依存していないか |
| ロール完全性 | 全ロールに責務・非責務・ライフサイクルが定義されているか |
| スキーマ整合 | 定義されている概念がスキーマに反映されているか |
| ドキュメント鮮度 | 最新バージョンの変更が既存ドキュメントに反映されているか |

## v1.8 リリースゲート（案）

### Gate 1: v1.7 Integrity Is Preserved

- AI Orchestrator ガバナンスモデルが後退していない
- 並列実行モデル・スレッド分離ルールが維持されている
- State Ownership モデルが整合している

### Gate 2: AOF-Native Task Management Is Defined

- `.aof/tasks/` 構造とライフサイクルが docs に定義されている
- タスク起票者と遷移権限が明示されている
- `schemas/aof-task.schema.json` が追加されている

### Gate 3: Alignment Pulse Task Triage Is Defined

- Task Triage が Alignment Pulse の確認項目として定義されている
- Orchestrator から Human への問い掛けパターンが定義されている

### Gate 4: Archivist Scope Is Extended

- Archivist がタスクアーカイブを担うことが定義されている
- Archivist のトリガー条件が定義されている

### Gate 5: .aof/ Directory Structure Is Canonical

- `docs/aof-directory-structure.md` が作成されている
- 全ディレクトリの用途・保持ルール・廃棄ルールが定義されている
- `schemas/aof-goals.schema.json` が追加されている

### Gate 6: Framework Self-Audit Is Defined

- `docs/framework-self-audit.md` が作成されている
- 自己監査のスコープ・トリガー・出力形式が定義されている
- 最初の自己監査が実施された証拠がある
