# v1.3 Operating Model Validation

`v1.3` の operating model を、現実に scope が変わる task で簡潔に検証する。

対象は、`ひまつぶしダンジョン` のように、軽い企画から世界観拡張へ drift しうる task とする。

## Validation Target

確認したいのは次の 4 点だけである。

1. discovery が初動を速くするか
2. sizing から initial team が出るか
3. operating goal と autonomy boundary で確認回数を減らせるか
4. drift が出たとき、次の判断へ戻れるか

## Scenario

### Initial Request

`ひまつぶし用のダンジョンゲームを考えたい`

### Later Drift

- 世界観をもっと強くしたい
- 画像や没入感も欲しい
- 企画だけでなく長編RPG寄りになってきた

## Discovery Acceleration Reading

### Current State Read

- surface request:
  - 軽いダンジョン案が欲しい
- highest uncertainty:
  - 軽量案で終えるのか、長編RPGへ広げるのか
- next evidence source:
  - 初回体験で何を魅力にしたいか

### First Hypothesis

- 最初の本当の need は「RPG全体」ではなく「初回プレイループの魅力を定めたい」かもしれない

### First Operating Goal

- `初回プレイループの魅力が伝わる1ページ企画に落とす`

### Recommended Initial Team Size

- reading:
  - `medium`
- initial team:
  - `Visionary`
  - `Builder`
  - `Guardian`
  - `Reviewer`

## Team Assembly Reading

- required actors:
  - `Visionary`
  - `Builder`
  - `Guardian`
  - `Reviewer`
- required skills:
  - framing
  - game-loop design
  - quality review
- owner:
  - `Builder`
- deferred roles:
  - brand / live-ops / worldbuilding specialist

## Goal Sizing And Autonomy Reading

- operating goal:
  - 初回プレイループ企画の確定
- success condition:
  - コア体験が短く説明できる
- autonomy budget:
  - wording / structure / small scope refinement は front AI が決めてよい
- escalation boundary:
  - 長編RPG化、workflow 変更、worldbuilding 主体化は確認対象

## Mission Control Reading

### Checkpoint 1

- current goal:
  - ループ企画の確定
- open signals:
  - none
- next checkpoint:
  - gameplay loop の説明可能性確認

### Drift Event

- signal:
  - `workflow mismatch`
- reason:
  - task が `micro-dungeon-design` から `longform-rpg-worldbuilding` に寄り始めた

### Response

- response:
  - `reframe`
- effect:
  - current operating goal を切り直す
  - team を `large` へ拡張するか判断する

## Result

この scenario では、`v1.3` operating model により次が言える。

1. 最初から大きな goal を抱えずに始められる
2. deferred roles を持つので初速を失いにくい
3. drift が出たら workflow mismatch として次の判断に戻せる
4. same vocabulary を short kickoff と longer-running work の両方に使える

## Verdict

`v1.3` の operating model は、軽い task から長い task へ変質する現実の request に対して、

- 初動を速くし
- 確認回数を減らし
- drift を再判断へ戻す

という 3 つの目的を、AOF の core を壊さずに満たせる。
