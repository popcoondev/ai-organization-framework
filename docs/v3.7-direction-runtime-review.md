# AOF v3.7 Direction Runtime Review

## Runtime Status At Direction Setting

Observed on the self-hosting `v3.6.0` runtime:

- `active_release` is aligned to `v3.6.0`
- `release-state-audit` passes
- `mission-control-benchmark` passes
- `organization-status` still reports the `v3.6` Mission Control goal as the current operating goal
- `roadmap-status` still returns a legacy alignment answer pointing to `TASK-009`
- `mission-control` still recommends `TASK-042` even though `v3.6.0` has already shipped

This means the release itself is truthful, but the post-release operating state is not.

## Core Diagnosis

`v3.6.0` improved operator visibility inside a release cycle.

Running the released runtime showed the next bottleneck:

- the runtime can represent a release accurately
- but it does not reliably advance its own planning state after that release completes
- roadmap guidance, Mission Control recommendation, and open-task truth can drift behind the shipped baseline

The next problem is therefore not richer visibility first.

The next problem is:

> How does AOF transition from a completed release into the next truthful operating state without manual cleanup?

## Direction Decision

The next release should be narrowed to:

`v3.7.0 = runtime situation assessment`

Theme:

- make the runtime diagnose its current real situation, truth conflicts, and best next operating move with low drift

## Required Outcomes

`v3.7.0` should introduce a bounded situation assessment layer that:

- detects when the current release work is already complete
- closes or reconciles stale implementation tasks that still point at shipped work
- updates operator-facing runtime guidance so Mission Control no longer recommends already-shipped work
- makes `roadmap-status` derive current alignment from present release truth rather than preserved historical decomposition
- keeps `next-value-slice`, `operating-goal`, and open release tasks consistent with the real post-release frontier
- exposes a runtime-native diagnosis packet instead of asking operators to infer the situation from a viewer alone

## What This Is Not

`v3.7.0` should not start with:

- richer ornamentation on top of Mission Control
- speculative live role-state expansion
- broader autonomy claims
- analytics-first work that depends on stale planning state

Those only become useful after the runtime can move cleanly from one completed release into the next truthful direction.

## Proposed Verification

The next release should prove at least:

- a shipped release no longer leaves its implementation task open
- Mission Control points to the next real release frontier after sign-off and release
- `roadmap-status` aligns with current release truth and current open release work
- post-release drift can be detected and repaired from canonical runtime artifacts

## Recommendation

Proceed with:

- `v3.6.0` = bounded Mission Control visibility
- `v3.7.0` = runtime situation assessment and roadmap truthfulness

This is the fastest path that preserves the integrity of:

- `v3.2.0` Need Validation
- `v3.3.0` discovery evidence contract
- `v3.4.0` release-state truthfulness
- `v3.5.0` command-routing truthfulness
- `v3.6.0` Mission Control visibility truthfulness
