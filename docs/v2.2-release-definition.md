# AOF v2.2 Release Definition

`v2.2.0` の主題は、AOF を organization model を持つ framework から、organization capability layer を明示的に表現・検証できる framework へ進めることである。

## Position

`v2.0.0` は AOF を別 project に導入できる entrypoint にした。  
`v2.1.0` は organization / council / team / role / contract / dependency を first-class にした。  
`v2.2.0` はその上に、skill / capability / resource / policy / analytics / learning-loop artifact を追加し、AOF を workforce-aware organization system へ進める版とする。

AOF の本質は依然として agent framework ではなく organization framework である。  
`v2.2.0` では agent execution の高度化より前に、organization が何をできるか、何を使えるか、どう統治されるか、どう計測されるかを first-class にする。

## v2.2 Goals

1. skill を first-class organization object として定義する
2. capability を typed registry として定義する
3. resource を first-class inventory として定義する
4. policy を machine-readable artifact として定義する
5. benchmark validation を CLI/runtime command として昇格する
6. metrics / analytics / learning loop を最低限 inspectable artifact にする
7. `v2.2` と `v3.x` の境界を明確化する

## Current Implementation Snapshot

As of `2026-06-13`, the following `v2.2` surfaces exist in the product:

- first-class schema artifacts for skills, capability registry, resource inventory, and policy set
- bootstrap integration that writes those artifacts during `init` and `upgrade`
- self-hosting sample artifacts under `.aof/` that exercise the capability layer
- `organization-verify` as a productized validation command for:
  - bootstrap / organization / orientation schema validation
  - capability-layer artifact schema validation
  - organization-to-capability-layer ref alignment
  - skill / capability / resource / policy cross-reference validation
  - contract artifact path existence checks

This means `v2.2` has already crossed from concept-only design into a verifiable capability-layer implementation.

## Out Of Scope

- dynamic resource scheduler
- automatic staffing and workforce balancing
- runtime policy enforcement engine
- adaptive organization split / merge execution
- full organization analytics dashboard
- closed autonomous learning loop
- autonomous AI workforce daemon

## Release Gates

### Gate 1: Skill Is First-Class

- AOF docs define a first-class skill model
- skill identity, owner, version, applicable roles, and expected outputs are representable
- skills are not treated as prompt-only conventions

### Gate 2: Capability Is Typed

- capabilities move beyond free-form strings
- capability categories are defined
- role / agent / skill relationships to capabilities are expressible

### Gate 3: Resource Inventory Exists

- resources are representable as first-class artifacts
- agents, tools, MCP servers, repositories, environments, and budget/token envelopes can be inventoried
- resource identity is separated from scheduling logic

### Gate 4: Policy Is Machine-Readable

- approval rules are representable
- escalation rules are representable
- delegation and access rules are representable
- policy is an artifact, not only prose

### Gate 5: Validation Is Productized

- benchmark validation exists as CLI/runtime commands
- organization validation is explicit
- decision validation is explicit
- contract consistency validation is explicit
- dependency reference validation is explicit
- artifact completeness validation is explicit

Current status:

- `organization-verify` now covers organization, dependency, contract-path, artifact-completeness, and capability-layer cross-reference validation
- `decision-verify` now covers decision-record schema and markdown/json pair validation as a standalone command
- decision-record validation also continues to exist through the runtime decision path
- a single unified benchmark command still remains future work

### Gate 6: Metrics And Analytics Are Inspectable

- basic metrics snapshots exist
- basic organization analytics snapshots exist
- outputs can be reviewed without a full analytics backend

Current status:

- `metrics-snapshot` writes a first-class metrics artifact from local AOF state
- `organization-analytics-snapshot` writes a first-class analytics artifact from local AOF state
- both are inspectable without any external telemetry backend

### Gate 7: Learning Loop Is Formalized

- outcome reports, self-audits, and improvement proposals connect explicitly
- learning-loop artifacts exist as inspectable records
- the loop is formalized without claiming autonomy

Current status:

- `learning-loop-snapshot` writes a first-class artifact joining latest outcome evidence, latest self-audit, current next value slice, and improvement focus
- the loop is inspectable and runtime-backed
- adaptive policy updates, autonomous re-orgs, and self-rewriting behavior remain deferred

### Gate 8: v3.x Boundary Remains Clean

- no claim of autonomous workforce management is made in `v2.2`
- dynamic allocation, enforcement, and adaptive optimization remain deferred to `v3.x`

## Intended Outcome

After `v2.2.0`, a project initialized with AOF should have not only an explicit organization model, but also an explicit capability layer:

```text
Mission
  -> Organization
    -> Council / Team / Role
    -> Skill / Capability / Resource / Policy
    -> Contract / Dependency / Metrics
    -> Validation / Analytics / Learning Loop
```

This should let an AI orchestrator reason not only about who decides and who executes, but also:

- what the organization knows how to do
- what resources it can legally and practically use
- what policy constrains behavior
- how organization health is measured
- how improvement signals are recorded

## v3.x Boundary

`v2.2.0` should stop at:

- represent
- validate
- inspect

`v3.x` should begin at:

- allocate
- enforce
- optimize
- learn adaptively
