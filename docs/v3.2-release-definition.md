# AOF v3.2 Release Definition

`v3.2.0` の主題は、AOF を
`raw need -> project execution`
の系から、
`raw need -> validated need -> project execution`
の系へ正式に進めることである。

## Position

`v3.1.0` までの AOF は、project が始まった後の governance と runtime evidence をかなり強く扱えるようになった。  
`v3.2.0` では、その前段にある

- そもそも何を project にするべきか
- stated request が本当に解くべき problem か
- no-project / defer / experiment-first を言えるか

を first-class release scope に含める。

ここでやるのは full Discovery Layer の完成ではない。  
やるのは、project creation の前に Need Validation gate を必須化し、
delivery runtime が wrong problem execution に強くなることを release claim にすることである。

## v3.2 Goals

1. Need Validation artifact family を canonical schema と CLI surface として確立する
2. framed session が `need-validation` stage を経ずに `planning` へ進めない runtime gate を導入する
3. `need-validation-advance` により、validated need と project charter がある場合だけ planning へ進める
4. `need-validation-benchmark` により `NV-001` から `NV-006` の benchmark surface を再実行可能にする
5. Discovery の role を project creation gate ではなく Need Validation input strengthening として再配置する

- automated market / user research execution
- generalized business-discovery autonomy
- always-on autonomous scheduler claims
- human-review removal claims

## Release Gates

### Gate 1: Need Validation Artifact Surface Exists

- problem statement schema and command exist
- value hypothesis schema and command exist
- alternative analysis schema and command exist
- experiment proposal schema and command exist
- project charter schema and command exist
- need validation record schema and command exist

### Gate 2: Pre-Project Runtime Gate Exists

- framed sessions stop at `need-validation`
- `council-exec` rejects `planning`, `proposal`, `review`, and `approval` before need validation advances the session
- a need validation decision record is emitted when clarification completes into a usable frame

### Gate 3: Controlled Advance Into Planning Exists

- `need-validation-advance` requires an approvable need validation record
- `need-validation-advance` requires a linked project charter
- only approved project-ready cases advance into `planning`
- operating goal projection is deferred until the planning transition

### Gate 4: Benchmark Surface Exists

- `need-validation-benchmark` evaluates `NV-001` through `NV-006`
- benchmark output is writable as a machine-readable artifact
- project-ready records and blocked records are both representable

### Gate 5: Docs And Runtime Story Agree

- core model describes Need Validation as mandatory pre-project gate
- operations model lists the Need Validation command family
- Discovery docs route handoff into Need Validation, not directly into delivery
- README read order exposes the Need Validation model

### Gate 6: Baseline Runtime Safety Remains Green

- `npm test -- --runInBand` remains green
- `node ./src/cli.js organization-verify --project .` remains green
- `runtime-loop-proof` still succeeds after the new gate is added
- `live-verify` still succeeds after the new gate is added

- `node ./src/cli.js live-verify --project . --provider mock --source-task-id TASK-011`

## Intended Outcome

After `v3.2.0`, AOF operator は次を release-grade contract として扱える。

```text
raw request
  -> framing
  -> need validation
  -> validated need
  -> project charter
  -> planning
  -> execution
```

この版の価値は、AI が execution governance だけでなく、
**wrong project creation を止める側の governance**
も持つようになる点にある。
