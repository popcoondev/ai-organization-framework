# AOF v1.7 Change Request

- 作成日: `2026-06-02`
- 作成者: レビュー分析（claude/framework-review-HCxns）
- 対象バージョン: `v1.7`
- 前提: `v1.6.0` リリース済み

## Position

`v1.6` が「live experience truth を統治する版」であるなら、  
`v1.7` は **「AI Orchestrator を第一級アクターとして定義し、並列 AI スレッド統治モデル（L3）を実現する版」** である。

### 核心的再認識

`v1.0` 〜 `v1.6` を通じて暗黙的に前提とされてきた構図を明示する。

```
Human（Approval / Review のみ）
        ↑↓ 会話・承認
AI Orchestrator（Claude Code / Codex 等）
  ├── .aof/ 状態管理
  ├── Value Alignment Loop 運用
  └── スレッド起動・集約・Human 向け要約
        ↓
  ├── Visionary ┐
  ├── Builder   ├── 並列（同一提案を独立評価）
  ├── Guardian  ┘
  ├── Discovery Scouts    ── 並列（ドメイン別）
  ├── Experience Steward  ── Builder と並列起動
  └── Value Slice Builders── 並列（スライス独立時）
```

`AOF の「組織」` とは人数ではなく、**1 人の Human が AI Orchestrator を介して複数の並列 AI スレッドを統治する構造**である。

### 実装上の重要発見

`council-exec` はすでに `--role` フラグで Seat 単位実行が可能である。

```
aof council-exec --session SESS-001 --stage approval --role Visionary
aof council-exec --session SESS-002 --stage approval --role Builder
aof council-exec --session SESS-003 --stage approval --role Guardian
```

Orchestrator が Agent tool でこの 3 コマンドを並列発行するだけで Council 並列実行は今日の実装で実現可能である。したがって **CLI コアへの変更は不要**であり、`v1.7` のランタイム変更はスキーマの追加フィールドのみとなる。

## 変更 1: 概念定義ドキュメント（新規）

### 1-1. AI Orchestrator ロール仕様（新規ドキュメント）

`docs/ai-orchestrator-role.md` を新規作成する。

定義すべき内容：

- Orchestrator の位置づけ（Human と並列 AI スレッド群の間に立つ AI エージェント）
- 責務
  - Human リクエストを Need / Intent / Context に変換する
  - North Star Goal・Operating Goal・Next Value Slice を `.aof/` に維持する
  - フェーズに応じて Layer 2 スレッドを起動・監視・終了する
  - 並列スレッドの出力を統合して Decision Record を生成する
  - Human が判断できる粒度に内部状態を要約して提示する
  - Alignment Pulse のタイミングで Human に再確認を求める
- **Orchestrator が持たない責務**（明示が重要）
  - Final Approval
  - Human 合意なしの North Star Goal 変更
  - Guardian Veto の無効化

### 1-2. 並列実行モデル（新規ドキュメント）

`docs/parallel-execution-model.md` を新規作成する。

定義すべき内容：

| フェーズ | 並列性 | 理由 |
|---|---|---|
| Discovery | 並列可 | ドメインが独立なら複数 Scout を同時起動 |
| Goal Layer 設定 | 逐次必須 | 全スレッドの共有錨。確定前に分岐不可 |
| Value Alignment Loop | 逐次必須 | Human 確認が前提。Orchestrator が問いを投げる |
| Clarification | Orchestrator 直接 | Human との Q&A。各アクター判断を内部集約して質問化 |
| Planning | 並列可 | 技術計画・ガバナンス計画は独立 |
| Build + Experience Steward | 並列 | Builder と Steward を同時起動 |
| Experience Validation | 並列可 | viewer fidelity / editor fidelity を別スレッドで評価 |
| Council Review | 並列（3 スレッド） | 同一提案を Visionary / Builder / Guardian が独立評価 |
| Human Decision | 逐次必須 | 集約後の判断は 1 点 |

- Clarification はアクターではなくフェーズである。Orchestrator が各アクター判断を内部で集約して Human への質問を設計する。独立スレッドは立てない。
- Experience Steward はスレッド間通信なしでビルダー成果物を受け取る。
  - デフォルト: Orchestrator → 完成成果物 → Experience Steward
  - Clarification 申請時: Orchestrator → 途中出力スナップショット → Experience Steward
- Council スレッド間の直接参照は禁止する。全通信は Orchestrator 経由とする。

### 1-3. ステート所有権マップ（新規ドキュメント）

`docs/state-ownership-model.md` を新規作成する。

定義すべき内容：

| ステート | 書き込み権限 | 確定条件 |
|---|---|---|
| North Star Goal | Orchestrator | Human との会話での合意後のみ |
| Current Operating Goal | Orchestrator | 前の Goal 完了を Orchestrator が判断した時、または Human 指示後 |
| Next Value Slice | Orchestrator（Builder 出力受け取り後） | Builder 完了後 |
| Recent Confirmation Window | Orchestrator | 各 Alignment Loop 後に自動更新 |
| Open Mismatch | Orchestrator（Experience Steward 出力受け取り後） | Steward 検出後に登録 |
| 各スレッドの評価内容 | Thread-local | Orchestrator 集約まで他スレッドから参照不可 |
| Decision Record | Orchestrator（集約後生成） | Human Approval で確定 |

## 変更 2: 既存ドキュメントの矛盾解消（更新）

### 2-1. `docs/council-execution-model.md`（C-1 / C-8）

**C-1 解消**: Orchestrator 自身の推論（Human との Q&A・状態管理）は single-instance でよい。Council Review は Orchestrator が 3 スレッドを並列起動するため multi-agent となる。「Orchestrator の実行パターン」と「Council Review の実行パターン」を分離して定義し直す。

**C-8 解消**: Council member の 2 モードを定義する。

- `Active Session mode`: 提案評価（現行の Council 動作）
- `Background Monitor mode`: フェーズ横断監視。特に Guardian の high-risk 案件監視に有効。Background Monitor mode で検出したリスクは Orchestrator 経由で Human へエスカレーション。

### 2-2. `docs/organization-sizing.md`（C-2）

sizing の軸に「Orchestrator 有無」と「Council 実行パターン」を追加する。

| サイズ | Orchestrator | Council 実行 |
|---|---|---|
| small | なし（Human 直接） | single-instance |
| medium | あり | hybrid |
| large | あり | multi-agent（並列） |
| extended | あり | multi-agent |

現行の role 数による定義と併用する。

### 2-3. `docs/value-alignment-loop.md`（C-3）

Alignment Pulse に 3 種のトリガー条件を追加定義する。

| トリガー種別 | 発火条件 |
|---|---|
| 時間ベース | N 分の wall-clock 経過 |
| マイルストーンベース | N スライスの Build 完了 |
| イベントベース | いずれかのスレッドが mismatch または escalation を発生させた |

N 値は v1.7 設計フェーズで確定が必要な開放問題として残る。

### 2-4. `docs/experience-steward-role.md`（C-4）

Orchestrator が Build 開始と同時に Experience Steward を起動することを明示する。Experience Steward のライフサイクルは Build フェーズ開始から Experience Validation フェーズ終了まで。

### 2-5. `docs/autonomy-and-goal-sizing.md`（C-5）

2 層の autonomy budget を定義する。

- **Orchestrator-Human 間 budget**: Orchestrator が Human 承認なしに進められる範囲（既存定義を継承）
- **Thread-Orchestrator 間 budget**: 各スレッドが Orchestrator 承認なしに進められる範囲（新規定義）

### 2-6. North Star Goal 所有権の明示（C-6）

`docs/value-alignment-loop.md` または `docs/state-ownership-model.md` に追記する。Orchestrator が North Star Goal を保持・更新するが、変更は Human との会話での合意が必要であることを明示する。

### 2-7. framing-only の再定義（C-7）

`docs/v1.4-release-definition.md` および usage level 関連ドキュメントを更新する。

framing-only を「Orchestrator なし（Human 直接）の形態」として再定義する。Orchestrator を導入した時点で partial-runtime 以上となる。

**注意**: これは v1.4 で定義した usage level の変更を伴う。後方互換性に注意が必要。

## 変更 3: ランタイム（スキーマのみ）

### 3-1. `schemas/aof-session.schema.json`

セッション親子関係を記録するフィールドを追加する。

```json
"orchestrator_session_id": {
  "type": ["string", "null"],
  "description": "親 Orchestrator セッションの ID。子セッションのみ設定する"
},
"child_session_ids": {
  "type": "array",
  "items": { "type": "string" },
  "description": "Orchestrator セッションが管理する子セッション ID のリスト"
},
"session_role": {
  "type": ["string", "null"],
  "enum": [
    "orchestrator",
    "council-visionary",
    "council-builder",
    "council-guardian",
    "discovery",
    "slice-builder",
    null
  ],
  "description": "並列実行時のこのセッションの役割"
}
```

### 3-2. `schemas/decision-record.schema.json` および `docs/decision-record-template.md`

並列実行のガバナンス品質を後から検証可能にするため 5 フィールドを追加する。

```json
"execution_pattern": {
  "type": ["string", "null"],
  "enum": ["single-instance", "hybrid", "multi-agent", null]
},
"parallel_thread_count": {
  "type": ["integer", "null"],
  "description": "並列実行したスレッド数"
},
"thread_isolation_mode": {
  "type": ["string", "null"],
  "enum": ["full", "partial", "none", null],
  "description": "スレッド間分離ルールの適用状況"
},
"independent_seats": {
  "type": "array",
  "items": { "type": "string" },
  "description": "独立実行した Seat 名のリスト。hybrid モード時に有効"
},
"execution_trace_refs": {
  "type": "array",
  "items": { "type": "string" },
  "description": "各スレッドの実行ログへの参照。独立性の事後検証に使用する"
}
```

`execution_trace_refs` は既存の `Protocol Thread ID` フィールドを複数スレッド対応形式に拡張したものとする。

## 変更しないもの

| コンポーネント | 理由 |
|---|---|
| CLI コマンド群 | `council-exec --role` で既に Seat 単位実行対応済み |
| セッション状態機械（12 ステート） | 変更不要 |
| council-exec コアロジック | 変更不要 |
| withSessionMutationLock | 各スレッドが独立セッションファイルを持つため競合しない |
| outcome-report / signal / answer / verify 系 | 変更不要 |
| visibility-serve | 変更不要（マルチセッション表示は v1.8 以降） |
| 既存テスト | 後方互換性を保つため変更不要。新規フィールドは optional とする |
