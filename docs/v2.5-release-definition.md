# AOF v2.5 Release Definition

`v2.5.0` の主題は、execution packet の次に来る governed allocation layer を first-class にすることである。

## Position

`v2.4.0` は role / team / council / lineage の execution contract を固定した。  
`v2.5.0` は、その execution に入る前に「なぜその resource をその role に割り当てるのか」「どの policy が何を要求するのか」を artifact として説明できるようにする版である。

ここでやるのは autonomous staffing claim ではない。  
allocation recommendation、policy evaluation、resource claim を schema-backed にし、reviewable な bridge layer を整えることである。

## v2.5 Goals

1. allocation recommendation を first-class artifact として記録できるようにする
2. policy evaluation を first-class artifact として記録できるようにする
3. resource claim を first-class artifact として記録できるようにする
4. workload-oriented allocation metrics を first-class snapshot に反映できるようにする
5. `v2.4` execution layer と接続しつつ、`v3.0` backend-neutral runtime claim を前倒ししない
6. existing verification and audit baseline を壊さずに governed allocation bridge を追加する

## Release Gates

### Gate 1: Allocation Plans Are First-Class

- `allocation-plan-record` exists
- allocation plan artifact is schema-backed
- target roles, candidate resources, recommended allocations, and risk notes are persisted

### Gate 2: Policy Evaluation Is First-Class

- `policy-evaluation-report` exists
- policy evaluation artifact is schema-backed
- overall outcome and per-policy results are persisted

### Gate 3: Resource Claims Are First-Class

- `resource-claim-record` exists
- resource claim artifact is schema-backed
- claim scope, claim status, approval policy refs, and justification are persisted

### Gate 4: Allocation Layer Is Coherent

- allocation plan, policy evaluation, and resource claim can reference one another coherently
- CLI accepts the documented flags for all three commands
- the runtime can explain recommendation -> policy review -> claim request as a traceable sequence
- `metrics-snapshot` can expose allocation volume, approval-required load, and open claim pressure

### Gate 5: Existing Runtime Baseline Remains Green

- `organization-audit` remains green
- `organization-verify` remains green
- `roadmap-status` remains usable and classifies v2.5 work correctly
- `npm test` remains green
- `npm run smoke` remains green

### Gate 6: Boundary Remains Honest

- `v2.5.0` does not claim autonomous resource scheduling
- `v2.5.0` does not claim runtime policy enforcement
- `v2.5.0` does not claim backend-neutral execution loop proof
- `v3.0` remains the first release for auditable end-to-end runtime proof

## Intended Outcome

After `v2.5.0`, AOF operator は execution 前の governed allocation layer を次の流れで扱える。

```text
allocation recommendation
  -> policy evaluation
  -> resource claim
  -> metrics snapshot / review load visibility
  -> review / approval decision
  -> execution layer
```

この版の価値は、自律配置ではなく、配置判断と制約判断を artifact として固定し、`v3.0` の runtime proof に渡せる状態を作ることにある。
