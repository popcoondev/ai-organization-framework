# AOF v4.0 Human Recognition Interface Spec

Date: `2026-06-19`

## Product Definition

`v4.0` is not merely a new recognition packet artifact.

`v4.0` should deliver a **Human Recognition Interface powered by the Human Recognition Packet**.

The product target is a one-screen operator surface where a human can instantly understand:

- who is in the organization
- which council exists
- which roles or characters are active
- what each character is working on
- what each character is currently recommending
- where the current work sits in the roadmap
- what changed over time
- what is blocked
- what should happen next
- whether the current answer is runtime-backed

The intended interaction model is:

- 2D game-like organization map
- council room and workforce room as separate readable spaces
- character cards with speech-bubble style recommendation surfaces
- roadmap sugoroku showing shipped trunk, active branch, and current frontier

This is a recognition interface, not decorative game UI.

Every visible state must derive from canonical runtime artifacts.

## Canonical Data Flow

```text
canonical runtime artifacts
  ->
Human Recognition Packet
  ->
Human Recognition Interface
  ->
operator recognition
```

The packet remains required, but it is an intermediate compression layer rather than the end product.

The interface must not show a generic loop unrelated to the current frontier.

The visible council / workforce / review / orchestrator sequence must trace back to the active frontier task and its supporting runtime artifacts.

## Character Model

### Councils

Councils should be shown as named rooms or seats, not generic labels.

Minimum visible councils:

- Product Council
- Architecture Council
- Operations Council

Runtime truth may also show a narrower effective council for the current loop, such as:

- planning council
- approval council
- review council

### Workforce Characters

Minimum visible characters:

- Visionary
- Builder
- Guardian

Optional future characters may be added, but `v4.0` must first make the base triad legible.

### Character Bubbles

Each speech bubble must derive from runtime artifacts:

- council seat bubble: `council_execution_runs[].steps[].result.output_summary`
- workforce bubble: `role-results[].recommendation` or `role-results[].rationale`
- review bubble: `council-review.decision_summary` or `recommendation`
- orchestrator bubble: `mission_control.next_action.recommended_action` or `outcome_reports[].note`

No invented bubble text is allowed.

If canonical artifacts do not provide a readable summary, the UI must show:

- `summary unavailable from canonical runtime artifacts`

instead of fabricating personality text.

## Derived Views

### 1. Mission Header

Must show:

- current mission
- current frontier task
- current branch / shipped trunk relation
- runtime-backed state

### 2. Roadmap Sugoroku

Must show:

- shipped releases
- active branch
- current frontier
- next target node if known

This is a branch-position view, not a gantt chart.

### 3. Organization Map

Must show:

- current council room
- runtime team room
- active characters placed by role
- whether a role is active, waiting, or completed

### 4. Character Status

For each visible character, show:

- role name
- current state
- what they are working on
- speech bubble summary
- artifact-backed provenance

### 5. Live Loop

Must show:

- council invocation
- workforce assembly
- council review
- orchestrator return
- next loop state

This panel is the minimum proof that the UI is showing organizational motion, not static state.

### 6. Timeline / What Changed

Must show:

- previous checkpoint
- current checkpoint
- bounded change summary
- why the change matters

### 7. Blockers / Next Action / Runtime Backing

Must show:

- active blocker if any
- next action
- runtime-backed status
- latest execution reference

## Truth Rules

The interface must obey these rules:

- no decorative character state without canonical artifact backing
- no speech bubble without artifact-derived summary
- no roadmap node without release/task/goal basis
- no blocker without evidence basis
- no next action without frontier or review basis
- no runtime-backed badge without execution-log and refreshed-artifact refs
- no current-frontier live-loop claim unless the frontier task is actually tied to a runnable session

## First Deliverable For v4.0

The first `v4.0` deliverable should be:

- one screen-level interface spec
- one canonical mapping from artifact -> packet -> interface
- one runtime-backed example chain using the current frontier

That deliverable is complete when a human reviewer can look at the interface spec and answer:

- what each panel means
- what artifact powers each panel
- where character speech comes from
- how roadmap position is derived
- how the current live loop is recognized
- how truth is preserved without decorative invention
