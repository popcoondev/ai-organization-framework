# AOF v4.0.1 Release Notes

- target version: `v4.0.1`
- current package version: `4.0.1`
- status: patch release candidate

## Summary

`v4.0.1` is the patch release that makes the `v4.0.0` Human Recognition Interface more readable at the exact point operators were getting stuck:

- what the current frontier task actually is
- what that task is supposed to do
- whether that task description came from the live task artifact or only from a weaker fallback path

`v4.0.0` established the one-screen Human Recognition Interface.

Running it against live runtime work exposed a narrower usability gap:

- the active frontier id was visible
- the next move was visible
- but the task's own scope was not explicit enough inside the hero surface

`v4.0.1` fixes that without changing authority, workflow boundaries, or the runtime contract.

## Highlights

### 1. Frontier Task Scope Is Visible In The Hero Surface

The Human Recognition Interface now surfaces the current frontier task scope directly in the hero area.

That means the operator can see not only:

- the frontier id
- the current stage
- the next action

but also:

- what the current task is actually trying to accomplish

### 2. Live Artifact Versus Fallback Source Is Distinguished

When the viewer can read the live frontier task artifact, task scope is shown as live artifact truth.

When that description is unavailable, the viewer falls back to the branch summary and marks that fallback path accordingly.

This keeps the interface honest about evidence strength instead of pretending every task summary is equally strong.

### 3. Visibility Fixture Coverage Now Exercises Real Task Description Reads

The visibility test fixture now includes a real `.aof/tasks/open/TASK-200.json` artifact with a task description.

This verifies that:

- `loadVisibilityViews()` can resolve the frontier task artifact
- `runtime_loop.current_frontier.task_description` is populated
- the Human Recognition Interface shell includes the task scope lane

### 4. Package Metadata Drift Is Corrected

`package-lock.json` metadata is now aligned with the release version again.

This patch does not change dependency contents, but it removes version drift between package metadata surfaces.

## Boundary

`v4.0.1` does not change:

- council authority
- runtime loop semantics
- Discovery / Need Validation boundaries
- roadmap direction
- viewer autonomy claims

It is a bounded patch focused on visibility truthfulness and operator comprehension.
