# AOF v3.4 Release Definition

`v3.4.0` の主題は、self-hosting runtime が
**現在どの release baseline を active と見なしているか**
を truthfully 保てるようにすることである。

## Position

`v3.3.0` は Discovery evidence を governed upstream contract にした。  
ただし self-hosting repo で runtime を実際に回すと、release 後の active refs に drift が残った。

- bootstrap metadata は古い `aof_version` を保持した
- governance-to-release contract は古い release definition を参照した
- roadmap surface は current release definition を古い値で返した
- organization mission も active baseline より古い framing を残した

したがって `v3.4.0` でやるべきことは autonomy の拡張ではない。  
やるべきことは **active release-state truthfulness** の導入である。

## v3.4 Goals

1. canonical active release manifest を first-class artifact として定義する
2. active release-state drift を runtime surface から検出できるようにする
3. bootstrap / organization contract を runtime-native command で refresh できるようにする
4. `organization-status` と `roadmap-status` が active manifest を観測できるようにする
5. `organization-verify` が active release alignment を cross-check できるようにする

## In Scope

- `docs/v3.4-release-definition.md`
- `docs/v3.4-release-checklist.md`
- `docs/v3.4.0-release-notes.md`
- `src/commands/release-state-refresh.js`
- `src/commands/release-state-audit.js`

- `schemas/aof-release-state-audit.schema.json`
- active release manifest integration in operator-facing status surfaces
- self-hosting repo refresh of active release refs

## Out Of Scope

- Discovery authority expansion
- Need Validation rule changes
- backend-specific orchestration claims
- broader external-reader automation
- human audit cost minimization beyond release-state truthfulness

## Release Gates

### Gate 1: Canonical Active Release Manifest Exists

- one active release manifest artifact exists under `.aof/context/active/`
- the manifest records release version, tag, release definition, release notes, checklist, roadmap, and release plan

### Gate 2: Drift Is Detectable

- a dedicated runtime command can detect missing manifest, stale bootstrap version, and stale governance release contract
- stale active refs are not silent

### Gate 3: Drift Is Repairable Through Runtime

- operator can refresh active release-state without manual JSON editing as the primary path
- refresh updates bootstrap and governance release contract consistently

### Gate 4: Operator Surfaces Converge

- `organization-status` can expose active release manifest information
- `roadmap-status` can resolve current release definition through active manifest
- `contract-register` and `organization-verify` converge on the repaired active release baseline

### Gate 5: Boundary Remains Honest

- `v3.4.0` does not expand autonomy claims
- Discovery remains evidence
- Need Validation remains authority
- backend-neutral positioning remains unchanged

## Verification Snapshot

`v3.4.0` should be checked at least by:

- `node --test test/runtime.test.js`
- `node ./src/cli.js organization-verify --project .`
- `node ./src/cli.js release-state-audit --project .`
- `node ./src/cli.js organization-status --project .`
- `node ./src/cli.js roadmap-status --project .`
- `npm run smoke`

## Intended Outcome

After `v3.4.0`, the self-hosting runtime should be able to say, and verify, one coherent thing:

```text
this is the active release
these are the active release refs
these runtime surfaces agree
and drift can be detected and repaired through the runtime itself
```
