# AOF v3.9 Direction Runtime Review

Date: `2026-06-18`

## Runtime Finding

Running the released `v3.8.0` runtime showed that the operator path improved, but the visibility surface still fails one important real-world test:

> the operator can read the current answer, but still cannot feel progress, branch position, or change-over-time from the same surface.

Observed on the current runtime:

- `operator-brief` correctly points at `TASK-045`
- Mission Control correctly points at `TASK-045`
- the viewer still looks mostly static between frontier shifts
- progress inside a frontier is weakly visible
- time-sequenced change is weak
- tree position inside the larger roadmap is weak
- visual surface design itself is not yet diagnosed as a first-class runtime concern

## Core Diagnosis

`v3.8.0` solved compact answer compression.

The next bottleneck is no longer:

- "What is the current answer?"

The next bottleneck is:

- "How did we get here?"
- "What changed since the last check?"
- "Which trunk and branch of the larger roadmap is this work on?"
- "Is the surface itself failing to explain progress clearly?"

This means `v3.9.0` should not be framed as proof lookup alone.

It should be framed as a bounded operator progress layer below the brief.

It also means `v3.9.0` must not stop at packet generation alone.

If the operator still has to manually wire the viewer every time, the release does not create a clear enough upgrade in real use.

## Direction Decision

Proceed with:

`v3.9.0 = bounded operator progress and evidence drill-down layer`

Theme:

- move from compact answer to bounded answer-to-proof and answer-to-progress
- make that surface reachable in one runtime path, including viewer-session orchestration

## Required Outcomes

`v3.9.0` should introduce a bounded runtime surface that:

- lets the operator inspect why the current headline, blockers, and next action are true
- shows what changed since the last meaningful runtime checkpoint
- shows which release trunk and active branch the current frontier belongs to
- diagnoses when the visibility surface itself is too static or too opaque for operator use
- can launch or attach the viewer path as part of the runtime flow instead of leaving browser opening as a separate manual ritual
- stays derived from canonical runtime artifacts rather than becoming a second operator story

## What This Is Not

`v3.9.0` should not broaden into:

- decorative dashboards first
- analytics expansion first
- role-state spectacle
- autonomy claims justified by visibility polish

The release should stay narrow and operator-truthful.

## Proposed Verification

The next release should prove at least:

- one bounded drill-down packet exists below the brief
- one bounded progress packet explains what changed since the last checkpoint
- one bounded tree-position packet shows current release trunk and frontier branch
- one bounded runtime command can open or attach the operator viewer session from the same canonical visibility packet
- the viewer or operator surface can show these without requiring raw artifact reconstruction
- surface diagnosis can detect "too static to explain progress" as a runtime problem

## Recommendation

Interpret the next sequence as:

- `v3.8.0` = truthful operator briefing
- `v3.9.0` = proof + progress + branch position below that brief
- `v4.x` = only after this, broader observability or analytics expansion
