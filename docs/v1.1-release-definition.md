# AOF v1.1 Release Definition

AI Organization Framework を `v1.1` として更新・リリースするための基準を定義する。

この文書の目的は 3 つある。

1. `v1.0` の上に何を進化させるかを固定する
2. `measurement foundation` と `operational policy` の境界を明示する
3. `v1.1` に入れるものと `v1.2+` に送るものを切り分ける

## Position

`v1.1` は新しい hosted product ではない。  
`v1.0.0` の core specification と local runtime prototype の上に、

- process を観測できる
- outcome を書き戻せる
- runtime をいつ使うかの方針を持てる

状態を追加する minor release である。

言い換えると、`v1.0` が `Completion` の最初の公開版なら、`v1.1` は
`Completion` と `Success` の差を pilot で測り始めるための版である。

## v1.1 Goal

`v1.1` の主目標は次である。

- AOF session に process telemetry を残せる
- decision 後の actual outcome を session に還流できる
- `AOF-only` と `runtime-on` の使い分け方針を明文化できる
- Codex のような front AI に対する concurrency expectation を明確にできる

要するに、`v1.1` は analytics を完成させる版ではなく、
**後から比較・改善できる観測基盤を入れる版**である。

## In Scope For v1.1

### Measurement Foundation

- machine-readable process metrics
  - clarification round count
  - routing mode history
  - reopen count の元データ
  - stage transition history
- session schema extension for `stage_transitions[]`
- stage transition persistence during normal lifecycle movement

### Outcome Reporting

- `outcome-report` CLI command
- session への outcome result 書き戻し
- minimum outcome state
  - `success`
  - `partial`
  - `failure`
- observed timestamp と short note の保存

### Runtime Usage Policy

- framing-only で十分な案件条件の文書化
- runtime を起動すべき案件条件の文書化
- Codex/front-AI 向けの session execution guidance

### Concurrency Policy

- current runtime は `1 session = serial mutation` 前提であることの明文化
- same-session concurrent mutation の扱いを固定する
- minimum safety policy
  - reject or prevent same-session concurrent writes
  - do not claim safe parallel stage mutation before hardening exists

### Verification And Examples

- tests for measurement fields and outcome reporting
- smoke or fixture coverage for persisted telemetry
- examples/docs reflecting the new flow

## Out Of Scope For v1.1

### Advanced Analytics

- automated cross-session comparison dashboards
- artifact quality scoring
- outcome scoring heuristics
- causal claims about which policy is best

### Advanced Concurrency

- branch/merge session model
- multi-writer conflict resolution
- seat-level parallel mutation with safe merge semantics

### Productization

- hosted metrics backend
- team dashboard UI
- external BI / audit integrations

## Non-Goals

`v1.1` の non-goal は次である。

1. measurement data だけで business success を保証しない
2. 初回から完全な analytics layer を作らない
3. session 並列化を安易に許可しない
4. all-runtime mandatory 運用を強制しない
5. provider-specific observability の拡張を先に主目的にしない

## Deferred To v1.2+

### Measurement Expansion

- cross-session comparison reports
- clarification-heavy vs clarification-light rollups
- fast-track vs deep-path rework trend reports
- reopen/outcome correlation summaries

### Concurrency Expansion

- session lock beyond single-process local assumptions
- optimistic concurrency / revision checks
- append-only event log
- branch merge semantics

### Runtime Ergonomics

- runtime output to implementation-task translation layer
- richer front-AI starter prompts and workflow presets
- more condensed human-facing review artifacts

## Release Gates

### Gate 1: v1.0 Integrity Is Preserved

- `v1.0.0` core lifecycle still works
- `v1.0.0` release surface is not regressed
- existing templates and release docs remain coherent

### Gate 2: Measurement Fields Are Real

- `stage_transitions[]` is represented in schema
- runtime persists transition events during actual lifecycle movement
- clarification/routing/reopen telemetry is machine-readable

### Gate 3: Outcome Reporting Is Real

- `outcome-report` exists as a working CLI command
- it writes outcome state back into the session
- session validation still passes after outcome writeback

### Gate 4: Runtime Usage Policy Is Explicit

- docs clearly state when framing-only is enough
- docs clearly state when runtime should be used
- docs clearly state that same-session mutation is serial-only unless future hardening changes that claim

### Gate 5: Evidence Collection Works

- tests cover measurement persistence and outcome reporting
- smoke or fixture flow proves telemetry survives end-to-end
- pilot users can collect data without custom scripts

## Release Evidence

`v1.1` readiness を主張するときは、最低限次を示す。

1. current commit SHA
2. green local test result
3. green smoke result
4. example session showing `stage_transitions[]`
5. example session showing an `outcome-report` writeback
6. docs for runtime usage policy and concurrency policy

## Current Intended Shape

現時点での `v1.1` の intended shape は次である。

- `collect first, analyze later`
- concurrency はまず「許可」ではなく「制御」を優先する
- Codex/front-AI には `AOF + optional runtime` の運用方針を与える
- `Success` を直接保証せず、`Success` を追える session history を増やす

## Relationship To v1.0

`v1.0` は「公開して使い始められる最初の一貫版」だった。  
`v1.1` は「使った結果を比較し、運用方針を改善し始められる版」である。

したがって、`v1.1` の完了条件は新しい派手な surface を増やすことではなく、
**AOF の運用を測定可能にし、front-AI/runtime の使い分けを制度化すること**にある。
