# AOF v3.0 Release Definition

`v3.0.0` の主題は、AOF が backend-neutral organization runtime として実行を統治できることを、self-hosting repo 上の auditable proof loop で示すことである。

## Position

`v2.6.0` は runtime state を operator-facing visibility layer に自動投影した。  
`v3.0.0` は、その可視化された runtime 面の上で、framing から allocation、execution、review、outcome、next-step recommendation までを 1 本の contract-driven loop として証明する版である。

ここでやるのは autonomous daemon claim ではない。  
role, join, team, council, allocation, policy, outcome, learning-loop の artifact 群が backend-specific 実装に閉じずに連結できることを示す。

## v3.0 Goals

1. backend-neutral execution contract を role / join / team / council artifact で貫く
2. governed parent/child orchestration model を human-started parent Codex 前提で明示する
3. policy-aware allocation and execution approval を runtime proof loop に参加させる
4. outcome と self-audit が next-step recommendation に inspectable に接続されることを示す
5. self-hosting repo 自身で `runtime-loop-proof` を実行し、proof artifact を残せるようにする

## Out Of Scope

- always-on autonomous daemon execution
- backend-specific child launcher claims
- Codex-only or Claude-only orchestration lock-in
- adaptive split / merge / archive automation
- organization self-modification without explicit human approval

## Release Gates

### Gate 1: Parent/Child Orchestration Is Contracted

- `role-result-record` exists
- `role-join-record` exists
- `team-output-record` exists
- `execution-lineage` includes role join state

### Gate 2: Allocation And Policy Participate In Runtime

- runtime proof creates an allocation plan artifact
- runtime proof creates a policy evaluation artifact
- runtime proof creates a resource claim artifact
- allocation and policy artifacts are linked into execution and review evidence

### Gate 3: One End-To-End Loop Is Auditable

- `runtime-loop-proof` exists
- one proof run records framing, allocation, execution, review, outcome, and next-step recommendation
- proof output includes canonical refs for every major artifact transition

### Gate 4: Self-Hosting Repo Can Run The Loop

- root `.aof` contains the workflow/template manifest set needed by `run` and `council-exec`
- `runtime-loop-proof --project . --provider mock --source-task-id TASK-011` succeeds on the self-hosting repo
- `roadmap-status` points to `docs/v3.0-release-definition.md`

### Gate 5: Existing Runtime Baseline Remains Green

- `organization-audit` remains green
- `organization-verify` remains green
- `npm test` remains green
- `npm run smoke` remains green

### Gate 6: Boundary Remains Honest

- `v3.0.0` claims one auditable backend-neutral runtime loop, not solved workforce autonomy
- `v3.0.0` does not claim always-on scheduler autonomy
- `v3.0.0` does not claim broad multi-provider production parity beyond the contract layer

## Intended Outcome

After `v3.0.0`, AOF operator は self-hosting repo 上で次の流れを実行証拠として扱える。

```text
request framing
  -> allocation plan / policy evaluation / resource claim
  -> role results
  -> role join
  -> team output
  -> council review
  -> outcome report
  -> learning loop recommendation
  -> runtime-loop-proof artifact
```

この版の価値は、AOF が単なる organization description ではなく、backend lock-in なしに governed execution loop を統治できることを、artifact で証明する点にある。
