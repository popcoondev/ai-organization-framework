# AOF v1.7 Release Definition

AI Organization Framework を `v1.7` として更新・リリースするための基準を定義する。

`v1.7` の主題は、AOF における **AI Orchestrator** を第一級アクターとして明示し、  
Human と並列 AI スレッド群の間で state, review, and aggregation を統治する model を formalize することにある。

## Position

`v1.7` は「runtime を全面刷新する版」ではない。  
また「viewer を multi-session ダッシュボードへ拡張する版」でもない。

`v1.6` で value alignment と experience validation を固めたうえで、`v1.7` は次を明示する版である。

- Human は final approval だけでなく alignment owner でもある
- AI Orchestrator は Human と worker threads の間で state, routing, and aggregation を担う
- Council, Discovery, Steward, and Slice Builder は thread として起動できる
- ただし parallel execution は independent judgment を自動保証しない
- execution topology と independence strength は別に扱う
- session ownership と aggregation failure handling を formalize する

言い換えると、`v1.6` が「何を見て alignment を保つか」を固めた版なら、`v1.7` は  
**誰がその loop を運転し、どう複数スレッドを統治するかを固める版** である。

## v1.7 Goal

`v1.7` の主目標は次である。

- `AI Orchestrator` を first-class role として定義する
- `execution topology` を定義する
  - `human-direct`
  - `orchestrated-single`
  - `orchestrated-parallel`
- `independence strength` を topology や execution pattern と分けて定義する
- `parallel execution model` を phase ごとに formalize する
- `state ownership` と `aggregation failure handling` を formalize する
- session / decision log schema に親子セッションと execution trace refs を追加する

## In Scope For v1.7

### AI Orchestrator Role

- `AI Orchestrator` を role として定義する
- responsibility を定義する
  - Human request を `Need / Intent / Context` に変換する
  - `North Star Goal / Current Operating Goal / Next Value Slice` を維持する
  - phase に応じて child threads を起動、監視、終了する
  - thread output を統合して human-readable summary と decision artifact を作る
  - `alignment pulse` と repeated confirmation を運用する
- non-responsibility を定義する
  - final human approval の代替
  - human agreement なしの `North Star Goal` 変更
  - Guardian veto の無効化

### Execution Topology

- usage level と別軸で `execution topology` を定義する
- minimum topology は次とする
  - `human-direct`
  - `orchestrated-single`
  - `orchestrated-parallel`
- `framing-only` を topology の別名にしない
- usage level と topology の両立ルールを定義する

### Independence Strength

- `parallel` を即 `independent` とみなさない
- minimum claim levels を定義する
  - `none`
  - `partial`
  - `stronger-but-not-guaranteed`
- model diversity, provider diversity, and prompt isolation が independent disagreement claim にどう効くかを記述する

### Parallel Execution Model

- phase ごとの並列可否を定義する
- minimum reading は次とする
  - Discovery は条件付き並列可
  - Goal layer setting は逐次必須
  - Value Alignment Loop は逐次必須
  - Clarification は Orchestrator 集約
  - Planning は条件付き並列可
  - Build と Experience Steward は並列可
  - Experience Validation は条件付き並列可
  - Council Review は並列可
  - Human Decision は逐次必須

### State Ownership And Aggregation Failure

- state ごとの write authority を定義する
- minimum ownership subjects は次とする
  - `North Star Goal`
  - `Current Operating Goal`
  - `Next Value Slice`
  - `Recent Confirmation Window`
  - `Open Mismatch`
  - thread-local evaluation
  - decision record
- child session failure, partial completion, abandoned child session をどう扱うかを定義する

### Schema Extensions

#### Session Schema

- 親子関係と thread role の optional field を追加する
  - `orchestrator_session_id`
  - `child_session_ids`
  - `session_role`

#### Decision Record Schema

- governance quality の事後検証に必要な optional field を追加する
  - `execution_pattern`
  - `parallel_thread_count`
  - `thread_isolation_mode`
  - `independent_seats`
  - `execution_trace_refs`

## Runtime And Viewer Impact

### Runtime

`v1.7` では runtime の全面改修を主張しない。  
core runtime command の全面変更は out of scope とする。

この版で最低限必要なのは次である。

- schema extension
- orchestration model の formalization
- child session / aggregation を扱う将来実装の contract

したがって、**CLI core rewrite は required ではない**。  
ただし、Orchestrator 実装を実運用する場合の orchestration layer は将来必要になる。

### Viewer

`visibility-serve` の multi-session 可視化は、この版の required scope ではない。  
`v1.7` では state ownership と execution trace references を formalize し、viewer 拡張は後ろに送る。

## Out Of Scope For v1.7

- multi-session dashboard implementation
- automatic orchestration daemon
- thread budget auto-enforcement
- adaptive alignment pulse tuning
- L4 orchestrator-to-orchestrator coordination

## Non-Goals

1. `parallel` を independent review の保証として主張しない
2. Human を approval-only actor に矮小化しない
3. `framing-only` を無理に削除しない
4. CLI core を全面置換しない
5. viewer を即 multi-session UI に拡張しない

## Release Gates

### Gate 1: v1.6 Integrity Is Preserved

- value alignment loop remains coherent
- experience validation semantics are not weakened
- human authority remains explicit

### Gate 2: AI Orchestrator Role Is Explicit

- docs define responsibilities and non-responsibilities
- docs distinguish orchestrator from council seats and human owner
- docs connect orchestrator to value alignment and aggregation

### Gate 3: Execution Topology And Independence Strength Are Explicit

- docs define topology separately from usage level
- docs define independence strength separately from mere parallelism
- docs explain what cannot be claimed from single-model parallel execution

### Gate 4: Parallel Execution Model Is Explicit

- docs define per-phase parallel / sequential rules
- docs explain why Clarification and Human Decision stay centralized
- docs explain thread communication boundaries

### Gate 5: State Ownership And Failure Handling Are Explicit

- docs define write authority for major state classes
- docs define child failure / partial completion handling
- docs define how aggregated decisions become canonical

### Gate 6: Schema Extensions Are Coherent

- session schema adds orchestrator/child relationship fields compatibly
- decision record schema adds execution-trace fields compatibly
- old sessions and decision records remain loadable

## Deferred To v1.8+

- multi-session visibility dashboard
- orchestration runtime implementation with automated child session spawning
- topology-aware viewer surfaces
- adaptive thread budgets
- cross-orchestrator coordination

## Current Intended Shape

`v1.7` の重点は、AOF を「AI actor が何人いるか」の図式から、

- Human
- AI Orchestrator
- worker threads

の 3 層構造として定義し直すことにある。

したがって `v1.7` は、AOF の value alignment and experience-validation strengths を維持したまま、  
**orchestrated multi-session governance model を formalize する版** とする。
