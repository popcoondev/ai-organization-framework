# AOF v1.3 Release Definition

AI Organization Framework を `v1.3` として更新・リリースするための基準を定義する。

この文書の目的は 3 つある。

1. `v1.2` の concept layer の次に、何を operating model として固定するかを明示する
2. Discovery を「質問入口」ではなく「初動加速と team assembly の起点」に再定義する
3. `v1.3` に入れるものと、以後の heavier automation を切り分ける

## Position

`v1.3` は analytics 拡張版でも hosted orchestration 版でもない。  
`v1.2` で discovery, knowledge state, organization sizing, interpretation lenses を固定したうえで、

- 初手の調査と現状把握をどう速くするか
- どの team を最初に立ち上げるか
- どの大きさの goal を front AI が確認なしで進めるか
- drift が起きたときに、どう次の判断へ戻すか

を operating model として固定する minor release である。

言い換えると、`v1.0` が公開版、`v1.1` が measurement foundation、`v1.2` が concept-layer 版なら、`v1.3` は  
**Discovery-driven team assembly と mission control を持つ版** である。

## v1.3 Goal

`v1.3` の主目標は次である。

- Discovery を初動加速フェーズとして定義する
- `scope sizing -> team assembly -> operating goal` の流れを固定する
- front AI が過剰確認なしで進めるための `autonomy budget` と `escalation boundary` を持つ
- workflow mismatch や priority drift を受けて再判断へ戻る運用ループを定義する

要するに、`v1.3` は concept をさらに増やす版ではなく、  
**AOF が初動を速くし、途中のズレを次の判断へ戻す operating model を持つ版** である。

## In Scope For v1.3

### Discovery As Acceleration

- Discovery を `Unknown -> Question -> Answer -> Insight -> Need` のみで終わらせず、初動設計フェーズとして定義する
- minimum discovery outputs:
  - current state read
  - `known / unknown / assumptions / confidence`
  - first hypothesis
  - first operating goal
  - recommended initial team size
- 調査優先順位と最初の一手を Discovery の成果物として扱う

### Discovery-Driven Team Assembly

- `Facilitator` が sizing 結果から初期 team を組む rule を持つ
- Actor を静的辞書としてではなく、mission に応じて assemble されるものとして説明する
- minimum assembly outputs:
  - required actors
  - required skills
  - owner
  - optional reviewers

### Goal Sizing And Autonomy Budget

- 大きすぎる request を front AI が進められる operating goal に切る rule を持つ
- minimum goal contract:
  - operating goal
  - success condition
  - autonomy budget
  - escalation boundary
- 「ユーザーが常時細かく指示する前提」を外した運用 guidance を持つ

### Mission Control

- minimum per-turn control structure:
  - current goal
  - owner
  - next checkpoint
  - open signals
  - deferred work
- 記録のための記録ではなく、「次の一手を変える control loop」として説明する

### Drift Detection And Reframe Loop

- minimum drift classes:
  - workflow mismatch
  - user-driven priority drift
  - scope expansion
  - output quality drift
- drift が起きたときの response:
  - continue
  - reprioritize
  - re-scope
  - reframe

### Documentation And Validation

- README か companion docs で、`Discovery -> Team Assembly -> Mission Control` の流れを明確化する
- at least one concise validation artifact で、「初速」「過剰確認の抑制」「drift loop」の観点を説明できるようにする

## Explicit Exclusions From v1.3 Core

`v1.2` までに作られたもののうち、`v1.3` では core gate に直接入れないものを次のように扱う。

### Reference Only

- `Interpretation Lenses`
- cross-domain minimal-model validation

これらは `v1.3` でも有用な背景資料だが、初動加速と mission control を主張するための core gate にはしない。

### Still Preserved, But Not Expanded Here

- measurement dashboards
- verification archive / dashboard family
- heavier runtime observability

これらは壊さないが、`v1.3` の中心価値には置かない。

## Out Of Scope For v1.3

### Heavy Runtime Automation

- autonomous workflow switching in runtime
- automatic signal generation from model traces
- hosted mission-control service
- full orchestration backend

### Heavy Organization Formalization

- large actor taxonomy
- domain-specific role libraries
- enterprise staffing models
- regulated-industry governance packs

### Deep Analytics

- quality scoring heuristics
- cross-session causal claims
- outcome optimization dashboards

## Non-Goals

`v1.3` の non-goal は次である。

1. team assembly を巨大な role catalog にしない
2. autonomy を無制限自律にしない
3. drift detection を fully automated runtime capability と誤認させない
4. interpretation lens の追加自体を成果にしない
5. verification artifact 群をさらに増やすことを主目的にしない

## Deferred To v1.4+

### Runtime Enforcement

- workflow auto-switching
- signal auto-generation
- stronger runtime mission-control persistence
- automated reprioritization actions

### Role And Skill Expansion

- richer actor contract library
- domain skill packs
- role-specific policy packs

### Analytics Expansion

- discovery-speed comparisons
- autonomy-budget effectiveness reports
- drift frequency / recovery rollups

## Release Gates

### Gate 1: v1.2 Integrity Is Preserved

- `v1.2.0` concept layer remains coherent
- discovery / knowledge / sizing docs are still consistent
- `v1.3` prep does not retroactively bloat `v1.2`

### Gate 2: Discovery Is An Acceleration Model

- docs show discovery produces first actionability, not just questions
- discovery outputs include current-state read and first operating goal
- discovery / clarification boundary remains coherent

### Gate 3: Team Assembly Is Explicit

- docs define how sizing leads to initial team assembly
- required actors / skills / owner are part of the model
- assembly remains lightweight enough for front-AI use

### Gate 4: Goal Sizing And Autonomy Are Explicit

- docs define operating goal sizing
- docs define success condition, autonomy budget, and escalation boundary
- docs explain how this reduces unnecessary user confirmation

### Gate 5: Mission Control And Drift Loop Are Explicit

- docs define a minimum mission-control checkpoint
- docs define drift classes and their responses
- the loop from work to next decision is readable and non-contradictory

## Release Evidence

`v1.3` readiness を主張するときは、最低限次を示す。

1. current commit SHA
2. docs showing Discovery as acceleration
3. docs showing team assembly and goal sizing
4. docs showing mission control and drift loop
5. at least one concise validation artifact proving the operating model on a realistic task

## Current Intended Shape

現時点での `v1.3` の intended shape は次である。

- `discover first` の次は `assemble and control`
- 初速を上げるために、discovery は current-state read と first operating goal を出す
- 役割の辞書より先に、mission に応じた team assembly rule を持つ
- front AI の確認回数を減らすために、goal sizing と autonomy boundary を持つ

## Relationship To v1.2

`v1.2` は discovery と sizing の概念を固定した。  
`v1.3` は、その概念を実務上の初動へ接続する。  

したがって、`v1.3` の完了条件は新しい method lens を足すことではなく、  
**Discovery から team assembly と mission control へつながる operating model を固定すること** にある。
