# AOF v2.1 Release Definition

`v2.1.0` の主題は、AOF を installer/bootstrap 可能な framework から、project ごとの AI organization を明示的に設計・運用できる framework へ進めることである。

## Position

`v2.0.0` は AOF を別 project に導入できる product-grade entrypoint にした。  
`v2.1.0` はその entrypoint が生成する state に、organization model を first-class に加える。

AOF の本質は Agent Framework ではなく Organization Framework である。  
Agent は実行資源であり、中心概念は organization / team / role / council / contract / dependency / knowledge / metrics / lifecycle である。

## v2.1 Goals

1. AOF core model を AI Organization Operating System として明確化する
2. `.aof/organization.json` を first-class bootstrap artifact にする
3. Role と Agent を schema 上で分離する
4. Council / Team / Contract / Dependency / Knowledge / Metrics / Lifecycle を最小表現できるようにする
5. `aof init` と `aof upgrade` が organization manifest を生成・補完できるようにする
6. AI recognition packet に organization manifest を含める

## In Scope

- `docs/aof-organization-model.md`
- `schemas/aof-organization.schema.json`
- `.aof/organization.json` bootstrap generation
- project bootstrap manifest の `organization_ref`
- current core docs の Organization OS への更新
- release planning docs

## Out Of Scope

- full resource scheduler
- metrics auto-measurement
- dependency graph visualization
- automatic organization split / merge
- Claude workflows backend as formal runtime
- autonomous AI organization daemon

## Release Gates

### Gate 1: Organization OS Is The Core Definition

- `aof-core-model.md` defines AOF as AI Organization Operating System
- Agent is documented as execution resource, not core identity

### Gate 2: Organization Manifest Exists

- `schemas/aof-organization.schema.json` exists
- `.aof/organization.json` is generated by bootstrap

### Gate 3: Role And Agent Are Separate

- organization schema has `roles`
- organization schema has `agents`
- role assignments connect roles to agent / human / tool resources

### Gate 4: Governance Is First-Class

- council model is documented
- default `2_of_3` approval policy is represented as policy, not hard-coded law
- escalation path is represented separately from hierarchy

### Gate 5: Operational Objects Are Represented

- contracts are representable
- dependencies are representable
- knowledge owners are representable
- metrics are representable
- lifecycle state is representable

### Gate 6: Installer And Upgrade Preserve The Model

- `aof init` writes organization manifest
- `aof upgrade` preserves existing organization manifest when present
- `aof upgrade` creates missing organization manifest when absent

### Gate 7: Recognition Packet Includes Organization

- bootstrap docs list `.aof/organization.json`
- `project-bootstrap.json` includes `organization_ref`

## Intended Outcome

After `v2.1.0`, a project initialized with AOF has not only goals and tasks, but also an explicit organization model:

```text
Mission -> Project -> Organization -> Council / Team / Role -> Assignment -> Agent
```

This lets an AI orchestrator reason about who decides, who executes, who owns knowledge, what contracts bind teams, and where escalation goes.
