# AOF v3.5 Release Definition

`v3.5.0` の主題は、AOF CLI を
**毎回全文読まずに routing できる runtime-native command surface**
へ整理することである。

## Position

`v3.4.0` は self-hosting runtime の active release baseline を truthful に保てるようにした。  
ただし runtime を日常運用で使うと、別の friction が残った。

- CLI command 数は十分に多くなった
- AI は理解できるが、毎回 full reference を読むと context cost が大きい
- 人間にも `status / verify / audit / register` の役割差が直感的ではない
- 問題は capability 不足ではなく command routing 不足だった

したがって `v3.5.0` でやるべきことは command を増やすことではない。  
やるべきことは **CLI Context Efficiency Layer** を canonical にすることである。

## v3.5 Goals

1. CLI surface を `read / verify / write / execute / observe` taxonomy で分類する
2. canonical `command-registry` artifact を first-class にする
3. AI recognition packet に command routing summary を組み込む
4. operator-facing read surface から top commands を発見できるようにする
5. routing drift を dedicated audit で検出できるようにする

- new orchestration authority
- Need Validation rule changes
- Discovery authority expansion
- external automation claims beyond command routing
- backend-specific agent execution claims

## Release Gates

### Gate 1: Command Taxonomy Is Canonical

- one canonical command catalog exists in runtime code
- registry entries expose category and purpose
- all supported commands can be represented in the registry

### Gate 2: Registry Artifact Exists

- `.aof/command-registry.json` can be written and validated
- init / upgrade generate the registry artifact
- self-hosting repo contains the registry artifact

### Gate 3: Recognition Packet Can Route From Minimal Context

- bootstrap references the command registry
- project orientation includes command registry ref and routing summary
- top commands and runtime flow are available without rereading the full CLI reference

### Gate 4: Operator Surfaces Expose And Verify Routing

- `command-register` returns routing metadata
- `organization-status` exposes command surface summary
- `organization-verify` validates command routing alignment
- `command-routing-audit` detects routing drift

### Gate 5: Boundary Remains Honest

- `v3.5.0` does not claim broader autonomy
- the change is a context-efficiency and routing improvement
- Discovery remains evidence
- Need Validation remains authority

## Verification Snapshot

`v3.5.0` should be checked at least by:

- `node --test test/runtime.test.js`
- `node ./src/cli.js organization-verify --project .`
- `node ./src/cli.js command-routing-audit --project .`
- `node ./src/cli.js organization-status --project .`
- `node ./src/cli.js command-register --project .`
- `npm run smoke`

## Intended Outcome

After `v3.5.0`, the runtime should be able to say, and verify, one coherent thing:

```text
this is the command surface
these are the top commands
this is how to route without reading the whole CLI reference
and routing drift can be detected through the runtime itself
```
