# AOF v1.9 Release Definition

AI Organization Framework を `v1.9` として更新・リリースするための基準を定義する。

`v1.9` の主題は、`v1.8` で formalize した AOF-native project memory を  
**通常のライフサイクル進行の副作用として自然に更新される self-hosting runtime** に押し上げることにある。

## Position

`v1.8` が `.aof/tasks/`, `.aof/goals/`, `Recent Confirmation Window`, `Framework Self-Audit` を  
first-class artifact として定義した版であるなら、  
`v1.9` は **それらが manual follow-up ではなく lifecycle-native に維持される版** である。

重点は次である。

- main lifecycle commands から project memory を自動更新する
- `North Star Goal / Current Operating Goal / Next Value Slice` を self-hosting loop の中で保つ
- `Recent Confirmation Window` を repeated confirmation の canonical working set として運用する
- `Alignment Pulse` と `Task Triage` を repo 自体で継続運用する
- `Framework Self-Audit` を定期的な operating loop に組み込む

## v1.9 Goal

`v1.9` の主目標は次である。

1. AOF がこの repo 自体を self-hosting loop で回せること
2. project memory が main lifecycle commands の side effect として自然に更新されること
3. value alignment / task triage / self-audit が isolated artifact ではなく continuous loop になること
4. manual catch-up を減らし、AOF の operating-system claim を一段強めること

## In Scope For v1.9

### Lifecycle-Native Project Memory Updates

少なくとも主要 lifecycle command は、必要な project memory surface を自動更新する。

対象の例:

- `run`
- `answer`
- `outcome-report`
- `signal`
- `escalation-resolve`
- 必要に応じて `council-exec` or equivalent execution stage command

この版で重要なのは、**artifact を書けること** ではなく  
**普通に進行した結果として artifact が更新されること** である。

### Self-Hosting Goal Loop

`v1.9` では次の 3 層を self-hosting repo 自体に対して維持する。

1. `North Star Goal`
2. `Current Operating Goal`
3. `Next Value Slice`

更新ルール:

- `North Star Goal` は安定的で、人間合意なしに変更しない
- `Current Operating Goal` は work cycle の current anchor として保つ
- `Next Value Slice` は最も近い user-visible or operator-visible value を表す
- `outcome-report` 等の completion signal は `Next Value Slice` の完了投影に接続される

### Recent Confirmation Window As Working Memory

`Recent Confirmation Window` を repeated confirmation の canonical working set とする。

最低限:

- 直近数回の確認履歴を保持する
- 同じ core question の再確認を許容する
- alignment / mismatch / scale direction を保持する
- `answer`, `alignment pulse`, `interview loop` 相当の入口から自然に更新される

### Alignment Pulse As Continuous Operation

`Alignment Pulse` は occasional note ではなく、self-hosting repo に対する継続運用ループとする。

最低限含むもの:

1. value alignment check
2. task triage
3. stale task review
4. next value slice review
5. self-audit trigger review

### Framework Self-Audit As Operating Cadence

`Framework Self-Audit` は release-only ではなく、operating cadence の一部として扱う。

最低限:

- self-audit summary artifact
- open task creation when a gap is found
- relation to current operating goal
- human-readable review note

## Runtime Impact

### Runtime

`v1.9` では runtime への更新が中心になる。  
ただし full daemon や autonomous scheduler を必須にはしない。

必要なのは次である。

- lifecycle command からの project-memory write path
- repeated confirmation write path
- task / goal / self-audit artifacts の自然更新
- self-hosting loop に必要な最小 orchestration

### Viewer

viewer 拡張は useful だが `v1.9` の core gate ではない。  
`visibility-serve` は existing contract を維持すればよい。

ただし、`v1.9` で runtime-native に保つ memory surfaces が人から見えなくなるのは避けたい。  
したがって viewer 自体の大改修は不要でも、existing visibility contract から最低限次を読める状態を保つ。

- current goal layer summary
- open/stale task summary
- recent confirmation reference or equivalent alignment summary

つまり、`v1.9` は **runtime-first, viewer-second** の版である。

## Out Of Scope For v1.9

- full background daemon
- autonomous pulse scheduler
- multi-session memory dashboard
- GitHub Issues sync automation
- cross-repo orchestration
- advanced viewer task boards

## Non-Goals

1. `v1.8` の task memory model を壊さない
2. goal projection を急に別 canonical source にしない
3. viewer 拡張を runtime integration より先にしない
4. self-audit を release-only activity に戻さない
5. 人間確認が必要な authority boundary を弱めない

## Release Gates

### Gate 1: v1.8 Integrity Is Preserved

- task ledger, goal projection, archivist scope, and self-audit semantics are preserved
- self-hosting additions do not weaken `v1.8` memory guarantees

### Gate 2: Main Lifecycle Commands Update Project Memory

- at least the primary lifecycle commands update task / goal / confirmation artifacts as side effects
- update behavior is explicit in docs
- update behavior is covered by tests or equivalent operator verification

### Gate 3: Goal Loop Is Maintained In Self-Hosting Operation

- `North Star Goal`, `Current Operating Goal`, and `Next Value Slice` are all present
- role of each layer is explicit
- completion and next-slice refresh behavior are defined

### Gate 4: Recent Confirmation Window Works As Repeated Confirmation Memory

- repeated core questions are supported
- recent-window retention rule is explicit
- alignment / mismatch / scale direction are preserved in artifact form

### Gate 5: Alignment Pulse And Task Triage Are Continuous

- Alignment Pulse is defined as continuing operation, not just occasional review
- task triage is part of the pulse
- stale task handling is explicit
- relation between pulse outputs and recent confirmation memory is explicit

### Gate 6: Self-Audit Is Part Of The Operating Loop

- at least one self-audit evidence artifact exists under the new loop
- self-audit can create or refresh canonical AOF tasks
- human-readable summary of the detected gap exists

## Deferred To v1.10+

- scheduled pulse automation
- full self-hosting orchestration daemon
- multi-session visibility for memory/task surfaces
- advanced task/goal UI
- automatic stale-retire heuristics

## Current Intended Shape

`v1.9` は、AOF を

**“artifact definitions can be written”**

から

**“normal lifecycle progression naturally keeps AOF’s own memory fresh”**

へ進める版とする。
