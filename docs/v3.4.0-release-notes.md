# AOF v3.4.0 Release Notes

- target version: `v3.4.0`
- current package version: `3.4.0`
- status: released
- release note revision commit SHA: `tracked by tag v3.4.0`

## Summary

`v3.4.0` is the release that makes the self-hosting runtime truthful about its active release baseline after a real release.

`v3.3.0` strengthened upstream Discovery governance.  
Running that runtime on the self-hosting repo then exposed a different class of weakness:

- active release refs drifted across bootstrap, roadmap, and governance surfaces
- the runtime could still operate, but it could not state cleanly which release baseline was active

`v3.4.0` fixes that weakness with a canonical active release manifest, a drift audit surface, and a runtime-native refresh path.

## Highlights

### 1. Active Release Manifest Is First-Class

The self-hosting repo can now persist one active release manifest under `.aof/context/active/active-release-manifest.json`.

That manifest records:

- release version
- release tag
- release definition
- release notes
- release checklist
- roadmap
- release plan

### 2. Drift Is Auditable

`release-state-audit` adds a narrow runtime surface that checks:

- manifest presence
- referenced active release docs existence
- bootstrap version alignment
- governance release contract alignment

This release does not treat stale active refs as an invisible operator problem anymore.

### 3. Drift Is Repairable Through Runtime

`release-state-refresh` gives the operator a runtime-native path to repair active release refs without manual JSON editing as the primary workflow.

It updates:

- active release manifest
- `.aof/project-bootstrap.json`
- `contract-governance-to-release` inside `.aof/organization.json`

### 4. Operator Surfaces Now Expose Active Release Truth

`organization-status` and `roadmap-status` can now include active release information when a manifest exists.

`organization-verify` also checks active release alignment when the manifest is present.

## Boundary

`v3.4.0` does not claim:

- broader multi-agent autonomy
- Discovery authority expansion
- Need Validation relaxation
- backend-specific orchestration expansion

It claims a narrower but necessary thing:

**the self-hosting runtime can keep its active release baseline explicit, auditable, and repairable.**
