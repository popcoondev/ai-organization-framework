# AOF v2.0 Release Definition

`v2.0.0` の主題は、AOF を **self-hosting spec repo** から **別プロジェクトへ導入できる product-grade entrypoint** に進めることである。

## Position

`v1.x` 系は concept / runtime / cadence / orchestration を積み上げてきた。  
`v2.0.0` はそれを **数ファイルの正本 docs と installer/bootstrap** に整理し、実際に導入できる形へまとめる版とする。

## v2.0 Goals

1. docs を少数の canonical files に圧縮する
2. `aof init` を installer/bootstrap entrypoint として成立させる
3. `aof upgrade` を bootstrap migration entrypoint として成立させる
4. managed-project default を install 時点で固定できるようにする
5. AI recognition packet を first-class にする
6. `How do I start using AOF?` に 1 つの現実的な答えを与える

## Canonical Doc Set

`v2.0.0` では docs を次へ圧縮する。

- `README.md`
- `docs/aof-core-model.md`
- `docs/aof-operations-model.md`
- `docs/aof-project-bootstrap-model.md`
- `docs/quickstart.md`
- `docs/cli-reference.md`
- `docs/v2.0-release-definition.md`
- `docs/v2.0-release-checklist.md`
- `docs/v2.0.0-rc-draft.md`
- `docs/v2.0.0-release-notes.md`

## In Scope

- docs consolidation
- legacy doc removal
- bootstrap schemas
- `aof init` bootstrap command
- `aof upgrade` migration command
- acquisition/install flow documentation
- topology-aware bootstrap defaults

## Out Of Scope

- autonomous Codex startup bridge
- hosted orchestration service
- npm registry distribution as the only install path
- full project summarization automation

## Release Gates

### Gate 1: Canonical Docs Are Reduced

- canonical docs are reduced to the `v2.0` set
- old docs are removed from the active surface

### Gate 2: Installer Exists

- `aof init` exists as a bootstrap entrypoint
- it can write `.aof/` skeleton and bootstrap packet

### Gate 3: Upgrade Path Exists

- `aof upgrade` exists as a migration entrypoint
- it can rewrite bootstrap manifests to the current installer shape without changing topology
### Gate 4: AI Recognition Packet Is Explicit

- `project-bootstrap.json` and `project-orientation.json` are formalized
- goals and recent confirmation seed files are generated

### Gate 5: Managed-Project Default Is Clear

- `managed-project` is the documented default
- default write target is `aof/state`

### Gate 6: Acquisition Flow Is Concrete

- GitHub tag/release acquisition path is documented
- clone -> install -> link -> init path is documented

### Gate 7: Current State Reflects The New Focus

- AOF task / goal / self-audit point to bootstrap + 2.0 consolidation

## Intended Outcome

`v2.0.0` should make it realistic for a user to say:

```text
I want to use AOF on this repo.
```

and have the system answer with one concrete entrypoint:

```bash
aof init --topology managed-project
```
