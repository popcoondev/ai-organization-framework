# AOF v4.0 Release Definition

`v4.0.0` の主題は、
**Human Recognition Packet を intermediate layer として使いながら、runtime truth を one-screen Human Recognition Interface に変えること**
である。

## Position

`v3.9.0` で runtime は次を truthful にした。

- current answer
- what changed
- where in the tree
- bounded proof path
- one runtime path to the viewer

しかし `v3.9.0` を実際の viewer と runtime loop で使うと、次の gap が残った。

- human が一目で organization の全体状態を掴めない
- council / actor / judgment / assignment / review が別々に見えてしまう
- current frontier が live runtime loop に裏打ちされているのか、その場で判断しづらい
- packet が truth compression であって product surface ではないのに、surface より前に出て見えてしまう

したがって `v4.0.0` でやるべきことは packet の追加ではない。  
やるべきことは **canonical runtime artifacts から Human Recognition Packet を作り、それを one-screen Human Recognition Interface として提示すること** である。

## v4.0 Goals

1. a first-class Human Recognition Interface surface exists
2. artifact -> packet -> interface mapping is canonical and documented
3. runtime-backed fact is explicit, not implied
4. the current frontier loop is visible as council -> workforce -> review -> orchestrator
5. council / actor / speech / timeline / blocker / next-action panels are artifact-derived
6. the visible loop is frontier-specific rather than only a generic proof success
7. truth boundary remains honest: the interface does not become a second source of truth

## Out Of Scope

- decorative game UI without artifact backing
- synthetic character speech that is not derived from canonical artifacts
- autonomy expansion beyond current council / Need Validation boundaries
- plugin-heavy visualization work that does not improve recognition
- a separate viewer-only truth model

## Release Gates

### Gate 1: Human Recognition Interface Exists

- the runtime can show on one screen what is happening now
- the runtime can show where this work sits in the trunk / branch / frontier tree
- the runtime can show who is acting and who is judging
- the runtime can show what should happen next

### Gate 2: Canonical Mapping Exists

- artifact -> Human Recognition Packet -> interface mapping is documented
- character, roadmap, timeline, blocker, and next-action surfaces have named artifact sources
- the interface spec is canonical rather than implied by code only

### Gate 3: Runtime-Backed Fact Is Visible

- the interface shows whether the current answer is runtime-backed
- the interface shows last runtime execution time
- the interface shows command chain and refreshed artifact refs
- missing runtime backing degrades the claim visibly instead of staying silent
- generic proof success is not enough unless the proof still matches the visible frontier state

### Gate 4: Live Organization Loop Is Legible

- the interface shows council invocation
- the interface shows council seat sequence and judgment
- the interface shows workforce assembly and returned outputs
- the interface shows council review and orchestrator return
- the loop shown to the operator is traceable back to the active frontier task

### Gate 5: Human Recognition Beats Static Visibility

- the interface shows timeline / what changed
- the interface shows blockers and next action
- the interface shows the current frontier as a bounded branch above shipped trunk
- the release creates a visible recognition upgrade over the `v3.9` surface

### Gate 6: Boundary Remains Honest

- every visible state is artifact-derived
- Human Recognition Packet remains intermediate layer rather than authority
- Discovery remains evidence
- Need Validation remains approval authority
- decorative UI cannot substitute for frontier-specific runtime truth

## Intended Outcome

After `v4.0.0`, the runtime should be able to give the operator one coherent surface:

```text
this is the live mission
this is the current frontier
this is where it sits in the larger tree
these are the councils and actors currently involved
this is what they are recommending
this is what changed
this is what is blocked
this is what happens next
and this is whether the answer is genuinely runtime-backed
```
