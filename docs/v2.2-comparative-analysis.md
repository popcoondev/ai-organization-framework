# AOF Comparative Analysis For v2.2 Planning

Date: 2026-06-13

## Purpose

This document compares AOF v2.1 against three adjacent categories:

- Agentic Organization
- Multi-Agent Governance
- AI Workforce Management

The goal is to clarify:

- what AOF already has
- what AOF is still missing
- what is unique about AOF
- what should be handled in `v2.2`
- what should stay deferred until `v3.x`

## External Baseline

Recent agent systems and agent platforms increasingly expose the following as first-class concepts:

- agent skills, tools, memory, and MCP-connected resources
- multi-agent delegation, handoffs, subagents, or teams
- policy and permission controls
- tracing, observability, analytics, and usage monitoring
- long-running workflow state and resume/review loops

Representative signals from current primary docs:

- OpenAI Agents SDK centers on agents, tools, handoffs, guardrails, sessions, tracing, sandbox agents, and sandbox capabilities such as filesystem, shell, memory, and skills.
- Claude Code now documents skills, subagents, hooks, MCP, permissions, organization policy enforcement, analytics, agent teams, and channels.
- AutoGen positions itself as an event-driven framework for scalable multi-agent systems, with extensions for MCP, distributed agents, and code execution.
- CrewAI explicitly exposes agent capabilities, skills, knowledge, memory, flows, guardrails, observability, and enterprise team management.

## AOF v2.1 Position

AOF v2.1 is already strong in organization modeling.

It treats the organization, not the agent, as the core object:

```text
Mission
  -> Project
    -> Organization
      -> Council
      -> Team
        -> Role
          -> Assignment
            -> Agent / Human / Tool
      -> Contract
      -> Dependency
      -> Knowledge
      -> Metrics
      -> Lifecycle
```

That gives AOF a stronger governance and operating-model foundation than most agent frameworks, which usually start from agent, tool, and workflow abstractions first.

## Comparison Matrix

| Concept | What external systems commonly have | What AOF v2.1 already has | What AOF is missing | Recommendation |
| --- | --- | --- | --- | --- |
| Skill | Reusable workflow packages, prompt bundles, or specialist extensions attached to agents | Role / agent separation gives AOF a place to attach skills conceptually | No first-class skill object, schema, registry, ownership, versioning, or assignment rules | `v2.2` |
| Capability | Named capability inventories used for routing, tool access, and delegation | Agents have a simple `capabilities` array in organization manifests | No typed capability taxonomy, no role-capability matching, no capability validation, no capability-based staffing | `v2.2` |
| Resource | Tools, MCP servers, repos, sandboxes, environments, budgets, and execution surfaces | AOF docs already define resource conceptually, and agents/humans/tools exist as assignees | No first-class resource schema, inventory, access policy, availability state, or scheduling model | inventory in `v2.2`, scheduling in `v3.x` |
| Policy Engine | Guardrails, permission modes, allow/deny controls, org policy, enforcement hooks | AOF has councils, approvals, escalation, decision records, and policy language in docs | No explicit machine-readable policy object or runtime enforcement engine | policy model in `v2.2`, enforcement engine in `v3.x` |
| Metrics | Usage, cost, latency, quality, approval, adoption, throughput dashboards | AOF already models metric identity and ownership in the organization ontology | No automatic measurement, no rollups by team/role/council, no benchmark CLI emitting org metrics | metric contracts in `v2.2`, full measurement in `v3.x` |
| Organization Analytics | Team usage, velocity, queue health, review load, decision bottlenecks, workforce visibility | AOF has decision records, cadence artifacts, verify artifacts, and organization structure | No organization analytics layer that aggregates those artifacts into workforce insight | basic snapshots in `v2.2`, full analytics in `v3.x` |
| Learning Loop | Persistent memory, feedback loops, evaluations, self-improvement, policy-tuned adaptation | AOF already has cadence, self-audit, retire review, and decision history | No explicit learning loop model joining outcomes, metrics, policy updates, and organization changes | artifact loop in `v2.2`, adaptive loop in `v3.x` |

## What AOF Already Has

Compared with agent-centric systems, AOF already has unusually strong support for:

- mission-first organization design
- councils as explicit governance bodies
- team / role / assignment separation
- role and agent identity separation
- contracts as first-class operating artifacts
- dependency graphs between organization units
- knowledge ownership
- metrics ownership
- lifecycle state
- decision records and council approval evidence
- cadence and self-audit artifacts

This means AOF is not starting from zero in governance. It already has the skeleton of an operating system for organizations.

## What AOF Is Missing

The main gap is not governance intent. The main gap is operational expression.

The missing concepts are:

- a first-class skill model
- a richer capability model
- a first-class resource inventory
- a machine-readable policy model
- metric collection and rollups
- organization analytics artifacts
- a closed learning loop that changes organization behavior from evidence

In other words, AOF already knows who the organization is, but it is still weak at expressing what the workforce can do, what resources it can use, how policy is enforced, how performance is measured, and how the organization learns.

## AOF's Distinctiveness

AOF's uniqueness is not that it can run agents in parallel. Other systems already do that.

AOF's real distinctiveness is:

- organization-first ontology instead of agent-first ontology
- explicit governance objects instead of hidden prompt conventions
- contracts and dependencies as operating artifacts, not just implementation details
- decision approval traceability through ADR / decision records
- role and agent separation, which keeps workforce design stable even if models change
- self-hosting benchmarkability, where AOF can be used to plan and audit its own evolution

This is a stronger base for serious organization design than most current agent frameworks, which are better at execution primitives than at governance primitives.

## Recommended v2.2 Scope

`v2.2` should focus on making missing concepts representable, inspectable, and verifiable.

It should not jump straight to autonomous workforce management.

Recommended `v2.2` scope:

1. Skill model
   Add a first-class `skill` object or `skills` registry with:
   - `skill_id`
   - name
   - owner
   - version
   - applicable roles
   - required resources
   - expected outputs

2. Capability model
   Replace free-form capability strings with a typed capability registry:
   - execution capabilities
   - reasoning capabilities
   - review capabilities
   - domain capabilities
   - governance capabilities

3. Resource inventory
   Add a first-class resource artifact for:
   - agents
   - tools
   - MCP servers
   - repositories
   - environments
   - budgets and token envelopes

4. Policy model
   Introduce a machine-readable policy artifact for:
   - approval rules
   - escalation rules
   - resource access rules
   - allowed delegation patterns
   - review requirements

5. Benchmark validation CLI
   Promote the benchmark checks from manual evidence to runtime commands:
   - organization validation
   - decision validation
   - contract consistency validation
   - dependency reference validation
   - artifact completeness validation

6. Basic metrics and analytics snapshots
   Add simple rollup artifacts, not full telemetry:
   - task throughput by team
   - approval latency
   - contract coverage
   - dependency bottleneck count
   - unresolved escalation count

7. Learning-loop artifacts
   Formalize a minimal improvement loop:
   - outcome report
   - self-audit
   - policy update proposal
   - organization change proposal

The key principle for `v2.2` is:

> Make the missing concepts first-class and verifiable before making them autonomous.

## Implementation Progress Snapshot

As of `2026-06-13`, the following parts of that recommendation are already reflected in the product:

- skill model docs and schema exist
- capability registry docs and schema exist
- resource inventory docs and schema exist
- policy-set docs and schema exist
- bootstrap writes those artifacts during `init` and `upgrade`
- self-hosting `.aof/` artifacts now include capability-layer examples
- `decision-verify` validates committed decision artifacts as a standalone surface
- `metrics-snapshot` and `organization-analytics-snapshot` now emit inspectable local JSON artifacts
- `learning-loop-snapshot` now emits an inspectable local artifact joining outcome, audit, and improvement focus
- `organization-verify` validates:
  - bootstrap / organization / orientation schema alignment
  - capability-layer schema alignment
  - organization refs into skills / capabilities / resources / policies
  - cross-reference consistency across the capability layer
  - contract artifact path existence

This shifts `v2.2` from abstract planning into partial execution.

What remains open for `v2.2`:

- productized decision-record validation as a standalone verification command
- metrics snapshot artifacts
- organization analytics snapshots
- unified learning-loop artifacts that connect outcome, audit, and improvement proposals

## Recommended v3.x Scope

`v3.x` should be reserved for dynamic and adaptive behavior that depends on those new artifacts already existing.

Recommended `v3.x` scope:

1. Resource scheduling and staffing
   - dynamic assignment of agents/tools to roles
   - load balancing across AI and human seats
   - budget-aware task routing

2. Runtime policy enforcement engine
   - approval gating at execution time
   - delegation restrictions
   - capability-based access control
   - policy-aware retries and escalation

3. Organization analytics layer
   - dashboards across teams, councils, roles, and contracts
   - bottleneck detection
   - workforce health scoring
   - forecast and capacity planning

4. Closed learning loop
   - metrics feed back into policy proposals
   - recurring failures propose contract changes
   - sustained overload proposes re-org or split/merge
   - successful patterns promote skills or capabilities

5. Adaptive organization runtime
   - split / merge recommendations
   - temporary task forces
   - delegated sub-organizations
   - supervised autonomous operating loops

The key principle for `v3.x` is:

> Once AOF can represent and verify workforce concepts cleanly, it can start operating and adapting them.

## Proposed Release Boundary

The practical release boundary should be:

- `v2.2`: represent, validate, inspect
- `v3.x`: allocate, enforce, optimize, learn

That boundary matches the current maturity of AOF.

AOF is already ahead on organization ontology. It is not yet ready to claim full AI workforce management. The safest and strongest move is to make `v2.2` the release that closes the representation gap without over-claiming runtime autonomy.

## Suggested v2.2 Theme

Recommended theme:

> AOF v2.2: Organization Capability Layer

This would make `v2.2` the version that adds the missing operational vocabulary between organization design and autonomous organization runtime.
