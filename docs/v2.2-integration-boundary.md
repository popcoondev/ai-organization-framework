# AOF v2.2 Integration Boundary

## Decision

`v2.2` では skill / capability / resource / policy を `organization.json` に直接埋め込まない。

最初の integration boundary は:

- `organization.json` は canonical organization topology を保持する
- capability layer は独立 artifact として保持する
- `organization.json` からそれらへの reference を持たせる

## Why

この境界を採る理由:

- `v2.1` の organization schema を壊しすぎずに拡張できる
- capability layer の versioning を organization topology から分離できる
- `v2.2` を represent / validate / inspect の範囲に保ちやすい
- `v3.x` で scheduling / enforcement / optimization を追加するときにも artifact 分離を維持できる

## Recommended Shape

`organization.json` には次のような top-level refs を追加するのが第一段階として自然である。

```text
organization
  -> skills_ref
  -> capability_registry_ref
  -> resource_inventory_ref
  -> policy_ref
```

対応する canonical artifacts:

- `.aof/skills.json`
- `.aof/capability-registry.json`
- `.aof/resource-inventory.json`
- `.aof/policies.json`

## What Stays In organization.json

`organization.json` に残すべきもの:

- mission
- topology
- councils
- teams
- roles
- assignments
- agents
- contracts
- dependencies
- knowledge owners
- metrics
- escalation
- lifecycle

## What Moves To Capability Layer

独立 artifact 側で持つべきもの:

- reusable skills
- typed capability registry
- resource inventory
- policy rules

## Validation Consequence

`v2.2` の validation command は少なくとも次を検証する必要がある。

- ref が存在すること
- referenced artifact が schema-valid であること
- role / skill / capability / resource / policy の cross-reference が解決すること

## v3.x Boundary Protection

この設計は `v2.2` を artifact integration に留める。

次はまだやらない:

- automatic resource allocation
- runtime policy enforcement
- capability-driven routing optimizer
- adaptive org restructuring

これらは `v3.x` に残す。
