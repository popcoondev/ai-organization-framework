# AOF v3.0.0 Release Notes

- target version: `v3.0.0`
- current package version: `3.0.0`
- status: ready for final tag
- release note revision commit SHA: `to be fixed by the release tag`

## Summary

`v3.0.0` is the release that proves AOF can govern one real organization runtime loop instead of only describing organization state.

This release adds first-class proof for:

- framing
- governed allocation
- child role execution
- parent join
- team aggregation
- council review
- outcome writeback
- next-step recommendation

`v2.6.0` made runtime state operator-visible.  
`v3.0.0` makes the same runtime state participate in one auditable backend-neutral execution loop.

## Highlights

### 1. Runtime Loop Proof Becomes Canonical

`v3.0.0` adds:

- `aof runtime-loop-proof`

This command produces a proof artifact that links:

- session and decision refs
- allocation plan / policy evaluation / resource claim refs
- role result / role join / team output / council review refs
- execution lineage and learning loop refs

### 2. Parent Join Becomes A Runtime Surface

`v3.0.0` adds:

- `aof role-join-record`

This closes the gap between child role outputs and parent orchestrator judgment, so parent/child orchestration is represented as runtime evidence instead of only prose.

### 3. Self-Hosting Repo Can Execute The Proof

The repo root now contains the `.aof/aof.yaml` manifest, actor files, workflow file, and decision template assets required for `run`, `answer`, and `council-exec` to operate directly on the self-hosting project.

## Boundary

`v3.0.0` is a backend-neutral runtime proof release.

It still stops before:

- always-on autonomous daemon execution
- backend-specific launcher claims
- unconstrained self-modifying organization behavior
- broad workforce-management claims beyond the proven contract layer

## Operational Note

`TASK-028` closed the observed Node `v25.5.0` instability by reducing CLI startup import load through lazy command loading and by explicitly marking Node 22 as the CI-validated runtime lane.

`v3.0.0` therefore does not claim Node 25 parity. The honest support expectation remains: use Node 22 LTS for release-grade operation.
