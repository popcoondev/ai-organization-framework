# AOF v2.3 Operator Surface Definition

## Purpose

`v2.3.0` is the release where the current `v2.2` organization and capability model becomes operator-usable without requiring direct JSON inspection.

This document defines the minimum operator-facing surface that must exist before `v2.3.0` can be considered complete.

## Scope

`v2.3.0` is still inside the `represent / validate / inspect` boundary.

It does not add:

- execution packet runtime
- governed allocation
- runtime policy enforcement
- autonomous orchestration claims

It does add:

- readable organization inspection
- readable contract and dependency inspection
- readable decision and roadmap inspection
- one unified organization audit path

## Required Commands

### `aof organization-status`

Goal:

- summarize current organization, topology, goals, task counts, councils, teams, roles, and capability-layer presence

Minimum output:

- topology
- install mode
- north star
- operating goal
- next value slice
- councils
- teams
- roles
- open / done / archived / retired task counts
- capability / resource / policy artifact presence

Primary source artifacts:

- `.aof/project-bootstrap.json`
- `.aof/organization.json`
- `.aof/goals/*.json`
- `.aof/tasks/*`
- `.aof/skills.json`
- `.aof/capability-registry.json`
- `.aof/resource-inventory.json`
- `.aof/policies.json`

### `aof contract-register`

Goal:

- list declared contracts and show whether their referenced artifacts exist

Minimum output:

- contract id
- name
- owner ref
- contract type
- artifact ref
- artifact presence state

Primary source artifacts:

- `.aof/organization.json`

### `aof dependency-graph`

Goal:

- surface declared dependencies in a human-readable and machine-readable form

Minimum output:

- from ref
- to ref
- dependency type
- lifecycle state

Recommended views:

- flat table
- adjacency list
- optional graph-oriented JSON export

Primary source artifacts:

- `.aof/organization.json`

### `aof decision-register`

Goal:

- list decision records and show whether canonical markdown and JSON pairs are present and aligned

Minimum output:

- decision id
- title
- status
- canonical markdown path
- JSON path
- pair alignment state

Primary source artifacts:

- `.aof/decisions/`

### `aof roadmap-status`

Goal:

- show how current goals, open tasks, and release-planning documents line up against the active roadmap

Minimum output:

- current next value slice
- prioritized task ids
- roadmap phase focus
- bridge-release task mapping
- release-plan artifact refs

Primary source artifacts:

- `.aof/goals/*.json`
- `.aof/context/active/alignment-pulse.json`
- `.aof/tasks/*`
- `docs/vnext-roadmap.md`
- `docs/vnext-release-plan.md`

### `aof organization-audit`

Goal:

- provide one operator-facing entrypoint that bundles the current verification surfaces into a single integrity readout

Minimum checks:

- organization schema validation
- ref alignment across organization, skills, capability, resource, and policy artifacts
- contract artifact existence
- decision record pair and schema validation
- task lifecycle duplicate detection
- task id uniqueness across lifecycle directories

Expected behavior:

- return a compact summary for operator use
- optionally write an audit artifact for later review
- fail clearly when integrity is broken

This command should unify, not replace, lower-level verification commands such as `organization-verify` and `decision-verify`.

## Release Gate For `v2.3.0`

`v2.3.0` is complete only if all of the following are true:

1. An operator can inspect organization state from CLI surfaces without opening raw `.aof/*.json` files.
2. Contracts, dependencies, decisions, and roadmap state are exposed as first-class runtime views.
3. A unified organization audit surface exists and catches task lifecycle duplication in addition to current schema checks.
4. The `v2.2` capability baseline remains intact and no `v3.x` claims are introduced.

## Dependency Order

`v2.3.0` should be implemented in this order:

1. `organization-status`
2. `contract-register`
3. `dependency-graph`
4. `decision-register`
5. `roadmap-status`
6. `organization-audit`

The audit command comes last because it should consume the lower-level inspection logic rather than duplicate it from scratch.
